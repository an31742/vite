### **核心追问一：说说你对事件循环的理解**

这道题如果只回答“宏任务微任务”，最多及格。我们要从**浏览器架构**讲到**框架调度**。

## 1. 本质

本质是“**浏览器渲染引擎在单线程模型下，通过多队列异步调度机制，协调用户交互、JS执行、渲染绘制三者关系**”。

## 2. 为什么这样设计

-   **单线程的必要性**：JavaScript需要操作DOM，如果多线程同时改DOM，加锁的复杂度会让浏览器无法保证稳定性和性能。
-   **不这样设计会怎样**：如果没有异步队列，一个长任务会完全阻塞用户交互和页面渲染，导致页面卡死。
-   **为什么分宏任务和微任务**：
    -   **宏任务**（Task）：每个宏任务代表一次完整的、独立的事件循环Tick的开始。如 `setTimeout`、`I/O`、事件回调、`requestAnimationFrame`（在渲染前）、UI Rendering（宏任务之间）。
    -   **微任务**（Microtask）：在当前宏任务执行完后、渲染之前，**必须**清空的队列。如 `Promise.then/catch/finally`、`MutationObserver`。
    -   **设计意图**：微任务提供了**高优先级、紧急性**的任务调度，让开发者能够在当前任务完成后立即处理结果，而不需要等到下一次渲染。

## 3. 底层原理（浏览器架构视角）

**事件循环完整流程（一轮Tick）**：

```
① 从宏任务队列取一个任务执行
   ↓
② 执行过程中可能产生微任务，入队微任务队列
   ↓
③ 当前宏任务执行完毕
   ↓
④ 【微任务检查点】清空微任务队列（包括执行过程中新产生的微任务，直到队列空）
   ↓
⑤ 如果需要，执行 requestAnimationFrame 回调
   ↓
⑥ 浏览器判断是否需要渲染（rendering opportunity）
   ↓
⑦ 如果需要： Style -> Layout -> Paint -> Composite
   ↓
⑧ 回到 ①，取下一个宏任务
```

**关键细节**：
-   **`requestAnimationFrame`**：在渲染之前执行，但不是每个宏任务后都会渲染。
-   **微任务可以产生新微任务**：会立即执行，直到队列彻底清空。这意味着**微任务死循环会永远阻塞渲染**。
-   **Node.js事件循环**：分6个阶段（timers、I/O callbacks、idle/prepare、poll、check、close callbacks），每个阶段之间清空微任务。`process.nextTick`是比`Promise.then`更优先的微任务。

## 4. 高级回答模板（联系Vue调度）

“面试官您好，事件循环我理解它不仅是理论，而是直接影响了Vue的`nextTick`实现和React的调度器设计。

从浏览器架构看，JS主线程是单线程的，事件循环就是它的调度操作系统。一轮Tick里，先执行一个宏任务，然后清空所有微任务，再决定是否渲染。

微任务的设计太精妙了。因为微任务在当前宏任务结束、渲染前必定执行，这让它成为**保证状态一致性**的完美时机。Vue的DOM更新就是异步的，它把多次数据变化收集起来，放入一个微任务队列统一进行`patch`。所以我们在`nextTick`的回调里，能拿到更新后的DOM，因为`nextTick`本质就是把回调追加到微任务队列尾部。

我做流式输出时，也用这个原理做了优化。AI返回的Markdown每次收到chunk都触发渲染，如果直接用宏任务，用户会看到一顿一顿。我们用微任务批量更新，一帧内可能处理几十个chunk，渲染却只触发一次，性能稳定在60帧。

还有，`Promise`产生的微任务如果不加控制，比如递归`Promise.resolve()`，会制造微任务死循环，页面永远不会重新渲染，直接卡死。这是我在调一个AI对话卡顿问题时的真实教训。”

## 5. 高频追问

-   **追问**：`requestAnimationFrame`和微任务谁先执行？
-   **回答**：微任务先执行。微任务在宏任务结束后立即清空，`rAF`在清空微任务后、浏览器决定渲染时才执行。**如果这一帧浏览器决定不渲染，`rAF`也不执行**。

-   **追问**：为什么Vue的`nextTick`优先用`Promise`而不是`setTimeout`？
-   **回答**：`Promise`产生微任务，能在当前宏任务结束后、渲染前立即执行，让DOM更新更早被看到。`setTimeout`是宏任务，需要等下一次事件循环，中间如果还有其他宏任务，延迟会更大。

### **核心追问二：如何理解ES6中的Promise**

## 1. 本质

本质是“**一种对异步操作结果的规范化状态容器，通过状态不可逆和回调注册机制，解决了回调地狱和控制反转问题**”。

## 2. 为什么这样设计

-   **解决了什么**：
    1.  **回调地狱**：异步操作嵌套导致代码不可读。
    2.  **控制反转**：把回调的执行权交给第三方，信任问题（回调可能被多次调用、不调用、调用时机错误）。
-   **Promise的解法**：
    -   **状态机**：`pending` -> `fulfilled` 或 `rejected`，状态一旦改变就**永久固定**，回调执行次数确定。
    -   **Promise/A+规范**：提供统一的`.then()`链式调用，返回新的Promise，实现顺序扁平化。
-   **Trade-off**：
    -   引入了额外的对象创建和微任务开销。
    -   无法取消（直到`AbortSignal`出现）。
    -   异常吞噬：未处理的rejection可能被静默。

## 3. 底层原理（实现规范）

**Promise/A+核心状态机**：

```javascript
// 简化Promise实现核心逻辑
const PENDING = 'pending';
const FULFILLED = 'fulfilled';
const REJECTED = 'rejected';

class MyPromise {
  constructor(executor) {
    this.state = PENDING;
    this.value = undefined;
    this.reason = undefined;
    this.onFulfilledCallbacks = []; // 存储成功回调
    this.onRejectedCallbacks = []; // 存储失败回调

    const resolve = (value) => {
      if (this.state !== PENDING) return; // 状态不可逆
      this.state = FULFILLED;
      this.value = value;
      // 发布：执行所有注册的成功回调
      this.onFulfilledCallbacks.forEach(fn => fn());
    };

    const reject = (reason) => {
      if (this.state !== PENDING) return;
      this.state = REJECTED;
      this.reason = reason;
      this.onRejectedCallbacks.forEach(fn => fn());
    };

    try {
      executor(resolve, reject);
    } catch (e) {
      reject(e);
    }
  }

  then(onFulfilled, onRejected) {
    // 2.2.7 then必须返回一个新的Promise
    return new MyPromise((resolve, reject) => {
      if (this.state === FULFILLED) {
        // 当前已经成功，异步执行回调
        setTimeout(() => {
          try {
            const x = onFulfilled(this.value);
            resolve(x); // 简化版，A+规范里有复杂的resolvePromise处理
          } catch (e) {
            reject(e);
          }
        }, 0);
      } else if (this.state === REJECTED) {
        setTimeout(() => {
          try {
            const x = onRejected(this.reason);
            resolve(x);
          } catch (e) {
            reject(e);
          }
        }, 0);
      } else {
        // PENDING状态：订阅回调，等resolve/reject时再执行
        this.onFulfilledCallbacks.push(() => {
          setTimeout(() => {
            try {
              const x = onFulfilled(this.value);
              resolve(x);
            } catch (e) {
              reject(e);
            }
          }, 0);
        });
        this.onRejectedCallbacks.push(() => {
          setTimeout(() => {
            try {
              const x = onRejected(this.reason);
              resolve(x);
            } catch (e) {
              reject(e);
            }
          }, 0);
        });
      }
    });
  }
}
```

**三个核心点**：
1.  **状态不可逆**：`state !== PENDING`时直接return。
2.  **回调延迟绑定**：即使先执行`then`，后`resolve`，回调依然正确执行（发布-订阅模式）。
3.  **`.then`返回新Promise**：链式调用的基础，也是`async/await`的底层。

## 4. 执行流程

1.  `new Promise(executor)` -> executor立即同步执行。
2.  调用`resolve(value)` -> 状态变`fulfilled`，值存`this.value`，异步执行所有已注册的`onFulfilled`回调。
3.  `.then(onFulfilled)`调用：
    -   如果状态已`fulfilled`，立即异步执行`onFulfilled`。
    -   如果状态`pending`，将`onFulfilled`推入`callbacks`数组，等待执行。
    -   无论哪种，都返回一个新的Promise。

## 5. 高频追问

-   **追问**：`Promise.resolve().then(() => {}).then(() => {})` 每个`.then`有几个微任务？
-   **回答**：在V8里，`.then`注册回调时，如果Promise已经`fulfilled`，会创建一个微任务来执行回调。如果还没有，则等`resolve`时创建微任务。具体是每个`.then`至少产生一个微任务。这是V8的实现细节，规范只要求“异步执行”。

-   **追问**：Promise和async/await的关系？
-   **回答**：`async`函数返回的是Promise，`await`的语义是`Promise.then`的语法糖。`await`后面的代码，等价于`.then()`里的回调，都会被放入微任务。

### **核心追问三：作用域、作用域链、闭包**

这三者是一体的，要放到**执行上下文**和**词法环境**的框架下理解。

## 1. 本质

-   **作用域**：“**代码在编译阶段确定的、用于约束变量可访问性的规则区域**”。
-   **作用域链**：“**由多个嵌套作用域形成的一条从内到外的变量查找链表**”。
-   **闭包**：“**函数在定义时捕获了其外部词法环境，并在其他地方执行时仍然保持对该环境的引用**”。

## 2. 底层原理（V8视角）

**词法环境（LexicalEnvironment）**：
-   JavaScript使用**词法（静态）作用域**。函数的作用域在定义时就确定了，不是调用时。
-   每次函数**定义**，都会创建一条`[[Environment]]`内部属性，指向当时的外部词法环境。
-   每次函数**调用**，会创建一个新的函数执行上下文，它有一个`outer`指针，指向`[[Environment]]`。

**闭包的形成**：
```javascript
function outer() {
    let a = 1;
    return function inner() {
        console.log(a); // inner 记住了它出生时的 outer 环境
    };
}
const fn = outer(); // outer执行完，按理a应该被GC
fn(); // 1，但因为inner的[[Environment]]还引着a，所以a不释放
```

**闭包的本质**：是**词法环境和垃圾回收机制的“博弈”**。引擎发现内部函数持有对外部变量的引用，就不回收外部作用域。

## 3. 高级回答模板

“面试官，这三个概念在V8底层是围绕**词法环境**展开的。

作用域是编译时就确定的一套查找规则。JavaScript引擎在预解析阶段就会扫描`var`、`let`、`function`，确定每个执行上下文的词法环境结构。所以是静态的。

作用域链，就是执行上下文栈里，当前上下文找不到变量时，沿着`outer`指针往外层找的这条链表。

闭包，我理解它的本质是JS词法作用域在内存管理上的体现。函数一出生就通过内部属性捕获了外部的词法环境。即使外部函数销毁了，只要这个内部函数还活着，外部环境的变量就死不了。

这在实际开发中是把双刃剑。比如在我们的大屏项目里，用闭包做事件处理器的防抖，确实方便；但如果不小心在循环里创建了大量闭包，或者闭包引用了大对象不释放，直接就是内存泄漏。V8的垃圾回收是标记清除，闭包让一些变量始终被‘标记’为可达，所以排查内存泄漏，我习惯用DevTools的Heap Snapshot，专门看closure是否引用了不该引的东西。”

### **代码执行题：详细分析**

```javascript
function fn(a, c) {
  console.log(a); // ① [Function: a]  
  var a = 123;
  console.log(a); // ② 123  
  console.log(c); // ③ [Function: c]  
  function a() {}
  if (false) {
    var d = 678;
  }
  console.log(d); // ④ undefined （注意！） 
  console.log(b); // ⑤ undefined （注意！）
  var b = function () {};
  console.log(b); // ⑥ [Function: b]  
  function c() {}
  console.log(c); // ⑦ [Function: c]  
}
fn(1, 2);
```

**这道题有陷阱，原题给的注释有误。正确输出如下：**

## 执行流程分析

**阶段一：编译（创建执行上下文）**

1.  函数`fn`被调用，创建函数执行上下文。
2.  形参绑定：`a = 1, c = 2`。
3.  变量提升（`var`）和函数提升（`function`）：
    -   先处理`var`声明：`var a`, `var d`, `var b`。记录变量名，初始化为`undefined`。
    -   再处理`function`声明：`function a(){}`, `function c(){}`。这两个提升，会**覆盖**同名的`var`声明。
    -   **关键**：`function a(){}`提升后，会覆盖形参`a = 1`。
    -   `function c(){}`提升后，会覆盖形参`c = 2`。

此时词法环境：
-   `a`： `function a(){}`
-   `c`： `function c(){}`
-   `d`： `undefined` （`if(false)`不影响声明提升）
-   `b`： `undefined`

**阶段二：逐行执行**

1.  `console.log(a)` -> 输出 `[Function: a]`。 （不是形参`1`，已被函数声明覆盖）
2.  `var a = 123` -> 变量`a`赋值为`123`。
3.  `console.log(a)` -> 输出 `123`。
4.  `console.log(c)` -> 输出 `[Function: c]`。 （形参`2`被函数声明覆盖）
5.  `function a() {}` -> **已经提升过了，此时不执行任何操作**。这个位置不改变`a`的值（仍是`123`）。
6.  `if (false) { var d = 678 }` -> `var d`已提升，但赋值语句因为`if(false)`永远不会执行。**所以`d`仍是`undefined`**。
7.  `console.log(d)` -> 输出 `undefined`。 （原题写`678`是错的，这是高频陷阱！）
8.  `console.log(b)` -> 输出 `undefined`。 （原题写`function`是错的！`var b`提升后是`undefined`，还没执行赋值语句）
9.  `var b = function() {}` -> `b`赋值为匿名函数。
10. `console.log(b)` -> 输出 `[Function: b]`。
11. `function c() {}` -> 已提升过，不执行。
12. `console.log(c)` -> 输出 `[Function: c]`。

**正确输出：**
```
[Function: a]
123
[Function: c]
undefined
undefined
[Function: b]
[Function: c]
```

## 高频陷阱

-   **`if(false)`里的`var`也会提升**：很多人以为`d`会报错或不存在，实际上`var`无视块级作用域，声明照样提升。
-   **函数提升 > 变量提升**：同名时，函数声明会覆盖`var`声明。
-   **函数声明不覆盖赋值**：编译阶段函数声明提升了，但执行阶段遇到`var a = 123`会把`a`重新赋值为`123`。代码中后出现的`function a(){}`不会把`a`变回函数。
-   **`var b = function(){}`是赋值，不是函数声明**：只有`b`的声明提升，函数值在赋值那一刻才确定。

### **剩余题目面试官视角评分点**

-   **基本类型 vs 引用类型**：不要只说`typeof`，要讲**栈内存存值（基本类型）vs 堆内存存对象，栈里存引用地址**。`typeof null === 'object'`是历史遗留Bug。可以补充`Object.prototype.toString.call()`更准确。
-   **深拷贝与浅拷贝**：浅拷贝（`Object.assign`、`...`扩展运算符）只复制第一层。深拷贝：`JSON.parse(JSON.stringify())`的局限性（丢失函数、Date、循环引用），结构化克隆`structuredClone`，或者手写递归 + WeakMap解决循环引用。
-   **防抖和节流**：防抖是“连续触发只执行最后一次”（如搜索框输入），节流是“固定时间间隔执行一次”（如滚动加载）。要结合你的项目场景，比如大屏的resize用防抖，AG-Grid滚动加载用节流。
-   **具名插槽和作用域插槽**：具名插槽解决“多个插槽分发”，作用域插槽解决“子组件数据暴露给父组件的插槽内容”，核心是**数据流向反转**。讲你的低代码平台里，某个表格的单元格插槽，父组件拿到行数据自定义渲染。
-   **路由hash和history**：hash的`#`部分不发送服务器，纯前端模拟；history基于HTML5 History API，需要服务器配置回退（所有路径返回`index.html`）。SEO友好度不同。要讲在微前端里，子应用路由模式的选择和主应用如何协调。

### **双精度问题怎么解决**

这个值得单独说。

## 1. 本质

本质是“**IEEE 754双精度浮点数标准用64位二进制无法精确表示所有十进制小数，导致计算出现舍入误差**”。

## 2. 解决方案

-   **转为整数计算**：`(0.1 * 10 + 0.2 * 10) / 10`。
-   **`toFixed`做展示**：但注意`toFixed`的舍入规则有时也有问题。
-   **使用库**：`decimal.js`、`big.js`。在金融场景必须用。
-   **ES2020 `BigInt`**：只解决大整数问题，不解决小数。
-   **`Number.EPSILON`**：判断两个浮点数是否“足够接近”，如`Math.abs(0.1+0.2 - 0.3) < Number.EPSILON`。

## 3. 项目实战

“在我们中后台的财务报表模块，后端返回的都是分（整数），前端只是展示时分转元并`toFixed(2)`，计算都在服务端或前端统一转成分再算，这样完全避开浮点数问题。”
