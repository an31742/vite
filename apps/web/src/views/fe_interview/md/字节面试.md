### 3. SSE AI 数据对话如何实现：发送 fetch 请求，while 循环，text-decoder 解析，结束标识

这题是考察你对之前项目细节的掌握。

**回答**（带执行流程）：

> “我们的 AI 对话不使用原生 `EventSource`，因为需要主动中断和自定义请求头。实现流程是：
>
> ① **发送 fetch 请求**：设置 `Content-Type: application/json`，请求体带 `conversationId` 和 `messages`。关键在获取 `response.body`，这是一个 `ReadableStream`。
>
> ② **创建读取器**：`const reader = response.body.getReader();`
>
> ③ **初始化 TextDecoder**：`const decoder = new TextDecoder('utf-8', { stream: true });`  
> `stream: true` 很关键，它能处理跨 chunk 的 UTF-8 字节截断，避免中文乱码。
>
> ④ **while(true) 循环读取**：
> ```js
> while (true) {
>   const { done, value } = await reader.read();
>   if (done) break;
>   const text = decoder.decode(value, { stream: true });
>   // 解析 SSE 格式
>   buffer += text;
>   const lines = buffer.split('\n');
>   buffer = lines.pop() || ''; // 保留不完整的行
>   for (const line of lines) {
>     if (line.startsWith('data: ')) {
>       const data = line.slice(6);
>       if (data === '[DONE]') break; // 结束标识
>       const parsed = JSON.parse(data);
>       // 将 delta 拼接到 Markdown 渲染器
>       appendToContent(parsed.choices[0].delta.content);
>     }
>   }
> }
> ```
>
> ⑤ **停止生成**：调用 `reader.cancel()`，浏览器会向服务端发送 RST_STREAM 帧，服务端捕获后立刻停止 LLM 推理。这比直接关闭 `EventSource` 更优雅。”
>
> **追问**：如何处理 chunk 边界刚好截断在 `data: {...` 中间？  
> **答**：我们用 `buffer` 存不完整行，下次循环再拼接。只解析完整的 `\n` 分隔的行。

### 4. 如何上手 React，React 和 Vue 用法/生命周期区别，Fiber 原理

#### 如何上手

> “我是从 Vue 转 React 的，我的路径是：
> 1. 先理解 JSX 和 React 的**单向数据流**、**不可变数据**思维。
> 2. 从 Class 组件了解生命周期，再用函数组件 + Hooks 重写，体会 `useEffect` 如何合并了 `componentDidMount/Update/Unmount`。
> 3. 写了一个小型的中后台 Demo，用 React Router 和 Context 做状态管理，感受和 Pinia 的区别。
> 4. 读 React 官网关于 **Fiber** 和 **Reconciliation** 的文档，结合调度原理理解。”

#### 用法区别（核心）

| 维度 | Vue | React |
|------|-----|-------|
| **响应式** | 自动依赖追踪（Proxy），直接修改状态 | 手动调用 `setState` / `dispatch`，不可变数据 |
| **模板** | 模板语法，编译时优化 | JSX，纯 JavaScript，灵活性高 |
| **更新机制** | 组件级精确更新（靶向） | 从根 fiber 开始 reconciliation，通过 fiber 链表协调 |
| **逻辑复用** | Composition API / mixins | Hooks / HOC |
| **编译优化** | 静态标记、PatchFlags、靶向更新 | 编译器自动加 `memo`、`useMemo` 等（React 19 Forget） |

#### 生命周期区别

> - Vue2: `created`, `mounted`, `updated`, `destroyed`，加上 keep-alive 的 `activated/deactivated`。
> - React Class: `componentDidMount`, `componentDidUpdate`, `componentWillUnmount`。
> - React Hooks: `useEffect` 把挂载、更新、卸载合并为一个函数，通过依赖数组控制时机。Vue3 的 `watchEffect` 类似但依赖收集更自动。

#### React Fiber 原理（深度回答）

**本质**：Fiber 是 React 16 引入的一种**可中断的异步协调算法**，将 VDOM 的递归 diff 转为**可暂停、可恢复的链表遍历**，使得主线程可以分片处理，不阻塞高优先级的用户交互。

**为什么需要 Fiber**：
老的 Stack Reconciler 是递归的，一旦开始就要一次性完成，会长时间占用主线程，导致掉帧。Fiber 把渲染工作拆成一个个 **fiber 工作单元**，每执行完一个单元就检查是否有更高优先级的任务，若有则让出主线程。

**底层实现关键**：
1. **Fiber 节点**：每个组件实例对应一个 fiber 节点，形成**链表结构**：`child`（第一个子节点）、`sibling`（下一个兄弟节点）、`return`（父节点）。
2. **双缓冲**：有 `current` 和 `workInProgress` 两棵 fiber 树，交替完成更新。
3. **工作循环**：`workLoop` 循环，调用 `performUnitOfWork` 执行单个 fiber 的 `beginWork` 和 `completeWork`，完成后检查 `shouldYield()`，如果时间片用完就暂停。
4. **优先级调度**：通过 Scheduler 包实现，基于 Lane 模型管理优先级。

**一句话**：Fiber 将 VDOM 的递归 diff 变成了一个可异步中断的链表遍历，让 React 具备了时间分片和优先级调度能力。

### 5. HTTP 和 HTTPS 区别

不要只背“多了 S”，要从**协议栈、安全、性能**三维讲。

**本质**：HTTPS = HTTP + TLS（SSL），在 TCP 和 HTTP 之间插入 TLS 层，提供**加密、身份验证、数据完整性**。

**关键区别**：

1. **安全**：
   - HTTP 明文传输，中间人可窃听篡改。
   - HTTPS 通过 TLS 握手，用非对称加密交换对称密钥，之后用对称加密传输数据，防止窃听和篡改。

2. **端口**：HTTP 80，HTTPS 443。

3. **连接建立**：
   - HTTP 直接三次握手后发请求。
   - HTTPS 在 TCP 三次握手后，还需 **TLS 握手**（约2个RTT），涉及证书验证、密钥协商。

4. **性能开销**：
   - HTTPS 有额外的加密解密计算和握手延迟。
   - 优化手段：HTTP/2 强制使用 HTTPS，支持多路复用，综合性能反而可能更好。Session 复用、OCSP Stapling 可减少 TLS 握手开销。

5. **证书**：HTTPS 需向 CA 申请数字证书证明身份。

**项目落地**：我们的内网系统也逐步切换 HTTPS，因为即使内网也要防劫持。同时配置了 HSTS 头，强制浏览器走 HTTPS。
