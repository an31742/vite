### **核心追问一：流式输出如果遇到页面刷新会怎么样？如何处理？**

这道题是上一批SSE的实战延伸，问的是**健壮性**和**用户体验**的边界。

## 1. 本质

本质是“**HTTP流式连接的生命周期与页面生命周期解耦后的状态恢复问题**”。

## 2. 为什么会出现这个问题

- **原生行为**：页面刷新（F5）会**立即销毁当前页面所有JS上下文、DOM、网络连接**。正在进行的Fetch流（SSE的ReadableStream）会被直接切断，已接收但未持久化的内容全部丢失。
- **用户体验灾难**：用户正在看AI生成的长回答，不小心F5了，回来发现一片空白，要重新开始对话，这是不可接受的。

## 3. 解决方案（从简单到高级）

**方案一：会话级持久化 + 断点续传（推荐）**

这也是我们在AI项目中实际落地的方案。

1.  **服务端设计**：
    -   每个对话会话（conversation）有一个唯一ID。
    -   AI生成的每一条消息，在服务端都被缓存或持久化。生成过程中，内容实时写入缓存。

2.  **前端刷新前**：
    -   在生成过程中，利用 `beforeunload` 事件（虽然不可靠，但聊胜于无）或**高频自动保存**机制。
    -   更关键的：我们不是只保存最终结果。我们把**每次收到的delta（增量）都拼到当前消息对象里，并定期同步到localStorage或sessionStorage**。做一层前端兜底。

3.  **前端恢复后**：
    -   页面初始化时，检查是否存在**未完成的会话**。
    -   如果有，向服务端发一个**状态查询请求**：`GET /api/conversation/{id}/status`。
    -   服务端返回该消息的最新完整内容。如果服务端还在生成中，前端可以：
        -   **方案A（简单）**：直接拿最新内容渲染，然后通过新的SSE连接续订后续更新。
        -   **方案B（精细）**：拿回完整内容渲染，然后结束。如果流还在跑，你不需要让用户傻等，而是直接展示“生成中，请稍候刷新查看”。

**方案二：Service Worker + 后台上传（PWA级方案）**

-   更高级，把流式数据的接收放到Service Worker层，SW生命周期独立于页面。
-   页面刷新后，SW继续收数据并存入IndexedDB，页面回来时从DB恢复。这属于大厂的极致体验优化，一般业务很少用。

## 4. 高级回答模板

“面试官您好，这是个非常实战的问题。我们做AI对话流式输出时，用户刷新页面确实会导致流连接断开，已经生成到一半的内容也丢了。

我们的处理是**多级兜底**。首先在服务端，每条消息都跟对话ID绑定，并且生成过程中实时把内容写缓存。在前端，我们不只在最后成功才保存，而是在`reader.read()`循环里，每收到一个chunk并拼好当前消息后，就**增量地把消息对象序列化存到sessionStorage**做前端兜底。

这样页面刷新后，初始化流程会检查`sessionStorage`里是否有未完成的会话。如果有，就拿着ID去问服务端要这个消息的最新完整内容。服务端有就有，不会丢。如果服务端那边还在生成，我们至少能展示已经生成的半成品，并提示用户‘正在恢复连接’，体验上就不是一片空白了。这本质是把网络流这种短期连接，跟会话状态这种长期生命周期，通过持久化做了个解耦。”

### **核心追问二：ag-grid 在vue中如何将表格渲染的12秒处理成1.2s？**

这是性能优化的高光时刻，要答出**问题定位**、**根因分析**、**量化优化**三步走。

## 1. 本质

本质是“**打通从数据准备、组件更新、到浏览器渲染的全链路瓶颈分析**”。

## 2. 12秒慢在哪？（性能排查过程）

一个Vue里的ag-grid表格数据量大导致渲染12秒，通常不是单一问题，而是**三重叠加**：

1.  **数据层**：后端返回几万条数据，但每条数据对象巨大，前端拿到后还要做二次处理（如嵌套字段展平、格式化），这计算本身可能就耗时2-3秒，且在主线程。
2.  **响应式层（Vue的锅）**：这是最常见、最致命的。如果你把几万条数据直接扔进一个`ref`或`reactive`里，Vue会对**每一个对象、每一个属性**进行深度的响应式劫持（getter/setter或Proxy）。几万条 × 几十个字段 = 几十万次递归操作，这个过程纯纯在阻塞主线程。
3.  **渲染层**：ag-grid拿到数据要创建DOM，虽然它有虚拟滚动，但首次挂载仍需渲染初始视口内成百上千个单元格，每个单元格还可能有自定义渲染器。

## 3. 怎么优化到1.2s（逐层击破）

**第一步：砍掉Vue响应式劫持（最关键）**
-   **原理**：ag-grid本身是一个纯数据驱动的表格库，它不需要Vue的响应式系统去监听数据内部变化来触发更新。你大可以直接把原始数据给它。
-   **怎么做**：
    -   不使用`reactive`包裹数据。用`shallowRef`或者一个普通变量`let rawData = []`来存。
    -   传给`ag-grid`的`rowData`时，直接传`rawData`。或者使用`markRaw`标记数据，告诉Vue跳过响应式转换。这一刀下去，12秒可能直接掉到3-4秒。

**第二步：优化数据处理管道（Web Worker或前端计算下沉）**
-   **前端预处理**：如果后端返回数据格式不对，需要前端转一次。把这个转换放到**Web Worker**里，让主线程不卡。处理完后`postMessage`回来，一个简单的数组。
-   **或者**：推动后端直接返回ag-grid需要的格式，前端完全不做二次加工。这就是**数据聚合下沉**。

**第三步：ag-grid配置优化**
-   **第一个也是最重要的：设置 `rowModelType` 为 `'serverSide'` 或 `'infinite'`**。
    -   12秒说明你可能在用**全量客户端模式**。把几万条全扔给浏览器，不慢才怪。
    -   改为 `infinite` 模式，ag-grid内部只创建一小部分DOM节点，滚动时动态替换数据，这才是虚拟滚动的正确打开方式。
-   **关闭不必要的特性**：如`animateRows: false`减少动画计算。避免在`cellRenderer`里使用复杂Vue组件，改用纯函数或简单HTML字符串。

## 4. 执行流程与量化

1.  **原始流程**（12s）：拿到数据 -> Vue深度响应式劫持(6-7s) -> 传给ag-grid -> ag-grid尝试渲染全部DOM -> 浏览器layout/paint(2-3s)。
2.  **优化后流程**（1.2s）：Worker处理数据(0.5s) -> 主线程拿到普通数组 -> `shallowRef`持有，不劫持 -> 传给ag-grid(infinite模式) -> 快速挂载当前视口内单元格 -> 浏览器快速首次绘制。

## 5. 高级回答模板

“面试官，这个12秒到1.2秒的优化我们真真切切做过。问题的核心不在ag-grid本身，而在**Vue响应式系统对巨量数据的过度处理**。

我拿到这个性能问题时，先用Chrome Performance录制了一次加载过程。发现一个巨大的Long Task，火焰图里的调用栈全在`reactive`的递归里，占了6秒多。

找到根因就好办了。我们的数据主要用于表格展示，并不需要每个字段变更都触发UI反应。我做的第一个关键决定：**切断Vue响应式**。把数据从`reactive`里解放出来，改用`shallowRef`存，传给ag-grid时直接给原始数组。这一步就把时间缩到4秒以内。

第二步，和ag-grid官方最佳实践对齐，打开**infinite滚动模式**，让它在DOM层面也做虚拟化，减少首次挂载的单元格数量。

最后，把前端的数据格式化逻辑移到**Web Worker**里，解放主线程。三管齐下，最终稳定在1.2秒左右。这个案例也让我深刻理解：不是所有数据都需要响应式，前端框架的特性要用在刀刃上。”

### **核心追问三：keep-alive 的实现原理是什么？**

## 1. 本质

本质是“**一个内置抽象组件，通过内存缓存VNode和组件实例，实现组件切换时的状态保持和避免重新渲染**”。

## 2. 为什么这样设计

-   **解决问题**：比如Tab切换，每次切走一个Tab，它的组件实例就销毁了，里面的表单、滚动位置全没了，切回来要重新创建、重新请求数据，体验差、性能浪费。
-   **Trade-off**：牺牲**内存**来换**CPU和用户体验**。你缓存了组件，DOM不销毁，JS对象不销毁，内存占用会更高。所以你得通过`include/exclude/max`来控制。

## 3. 底层原理（源码级）

keep-alive的实现围绕三个核心：

1.  **抽象组件**：它自身不渲染任何DOM元素，它是一个函数式或抽象组件（在Vue3中是`ShapeFlags`标记的特殊组件），渲染的是它的插槽里匹配到的第一个组件子节点。
2.  **缓存存储**：内部维护一个`cache: Map<string, VNode>`和一个`keys: string[]`。
3.  **挂载/激活钩子**：
    -   首次渲染子组件时，正常走`mount`，并把VNode和组件实例放入缓存。
    -   当再次切换到已缓存的组件时，keep-alive不会走`mount`流程，而是触发组件的`activated`生命周期钩子（Vue2）或通过调度器重新激活（Vue3）。

**Vue3简版核心逻辑**：

```javascript
const KeepAliveImpl = {
  __isKeepAlive: true,
  setup(props, { slots }) {
    const cache = new Map(); // 缓存 <key, VNode>
    const keys = new Set();
    let pendingCacheKey = null;

    // 获取需要被缓存的子节点
    const instance = getCurrentInstance();
    const sharedContext = instance.ctx;

    // 核心渲染逻辑
    return () => {
      const rawVNode = slots.default(); // 获取插槽里的第一个孩子
      // ... 判断是否有多个根节点，略
    
      const vnode = rawVNode;
      const comp = vnode.type;
      const key = vnode.key == null ? comp : vnode.key;
      
      const cachedVNode = cache.get(key);
      if (cachedVNode) {
        // 命中缓存！直接把缓存的组件实例拿过来
        vnode.component = cachedVNode.component;
        // 标记为需要激活，避免重复走 mount
        vnode.shapeFlag |= ShapeFlags.COMPONENT_KEPT_ALIVE;
        // 更新 keys 顺序，用于 LRU
        keys.delete(key);
        keys.add(key);
      } else {
        // 未命中，放入缓存
        keys.add(key);
        // 如果超过 max，清除最久未使用的
        if (props.max && keys.size > parseInt(props.max)) {
          const pruneKey = keys.values().next().value;
          keys.delete(pruneKey);
          cache.delete(pruneKey);
        }
      }
      // 标记当前正在处理的 key，用于 activated 钩子触发
      pendingCacheKey = key;
      
      vnode.shapeFlag |= ShapeFlags.COMPONENT_SHOULD_KEEP_ALIVE;
      return vnode;
    };
  }
};
```

## 4. 执行流程

1.  组件A首次渲染：`cache`没命中，正常调用A的`setup`、`render`，挂载DOM。VNode被存入`cache`。
2.  切换到组件B：A的组件实例和它的真实DOM，被移出DOM树但**没有销毁**，实例仍在内存中。VNode被保存在`cache`里。
3.  切回组件A：keep-alive从`cache`取回A的VNode，它已经带有之前的`component`实例。
    -   **不会**再次调用`setup()`或`created()`。
    -   **不会**重新创建DOM。它从内存里取出之前的DOM片段，直接插入到正确的位置。
    -   触发`activated()`生命周期钩子，你可以在这里处理恢复时需要的逻辑。

## 5. 高频追问

-   **追问**：`activated`和`mounted` 谁先触发？
-   **回答**：首次进入keep-alive时，先`mounted`再`activated`。从缓存恢复时，只触发`activated`。

-   **追问**：LRU（最近最少使用）缓存淘汰策略怎么工作？
-   **回答**：当设置`max`时，每次命中缓存或新增缓存，都会把当前key设为最新。当超出上限，会删除最老的那个key。这是用内置的`keys`顺序或集合维护的。

### **剩余题目面试官视角核心评分点**

-   **Vue2与Vue3区别**：不要再背“Composition API、Fragment、Proxy”的概念。要讲**设计范式**的升级：从Options API的“按选项类型组织”到Composition API的“按逻辑关注点组织”，解决了mixins的命名冲突、来源不清。讲响应式从`Object.defineProperty`的“对象劫持”到`Proxy`的“代理”带来的根本变化（数组、动态属性、性能）。
-   **为什么Vue3可以有多个根节点**：**Fragment**。Vue2的`render`函数必须返回单个VNode，底层`patch`比较依赖单根。Vue3重写了`patch`，支持处理`Fragment`类型的VNode，可以直接`patch`一个子节点数组。这就是“多根”的本质。
-   **nextTick原理**：本质是“**将回调推迟到下次DOM更新循环之后执行的异步降级策略**”。优先使用`Promise.then` -> `MutationObserver` -> `setImmediate` -> `setTimeout`。在Vue内部，`nextTick`与组件的`update`调度器共享同一个微任务队列，确保你拿到的是已经更新后的DOM。
-   **Vue2编译原理**：`template` -> `parse`（AST）-> `optimize`（标记静态节点，diff时可跳过）-> `generate`（生成`render`函数代码字符串，如`_c('div', [...]， _v(...))`）。你要讲出**静态标记**的优化意义。
-   **data为什么是函数**：对象是引用，组件复用时多个实例会共享同一个data对象，导致状态污染。函数则每次创建实例都会返回新对象。不要说“Vue要求的”，要说“**这是JavaScript对象引用机制在组件复用场景下的必然设计**”。
-   **Vue生命周期请求放哪里为什么**：常规放`created`或`mounted`都行，但SSR场景下`mounted`不执行，所以请求建议放`mounted`（不阻塞首屏）或`created`（客户端）。我现在更多用`async setup`或者`onMounted`，配合suspense做请求。要体现出你知道SSR限制和请求时机对用户感知首屏速度的影响。
-   **直接使用vuex变量会怎样**：如果你没有遵循“在`computed`中读取状态，通过`commit/dispatch`修改状态”的规则，直接在模板里用`$store.state.xxx`，响应式依赖是能收集到的。但问题是，修改必须通过mutation，否则会失去状态的可预测性和DevTools追踪。而且直接赋值不具备响应式。要从**单向数据流**和**可预测状态管理**的架构理念去回答。

### **一纸通分屏处理**

这题是典型的项目实战题，体现你的架构设计能力。

**本质**：本质是在单一文档流中，建立多个独立的、可拖拽调整大小的空间视口，并保持同步状态。

**架构设计**：
1.  **布局层**：采用CSS Flexbox或绝对定位，中间一个可拖拽的分隔条。监听`mousedown/mousemove/mouseup`动态修改左右两部分的`flex-basis`或`width`。
2.  **逻辑层**：一个核心状态表示当前分屏位置（百分比），组件卸载前存到`localStorage`，下次恢复。
3.  **数据同步**：如果左右屏展示的是同一个数据（如文档的一级标题、二级内容），它们应该共享同一个响应式数据源（一个Pinia Store或组合函数）。一侧修改，另一侧通过`watch`或直接响应式绑定自动更新。

**进阶追问**：如何支持嵌套分屏？
-   **回答**：将分屏组件设计成递归的。每个分屏面板可以是另一个分屏容器。这就是IDE的编辑器模式，本质上是一个树形布局结构。
