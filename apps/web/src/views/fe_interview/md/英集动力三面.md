### 1. 数字孪生项目是使用的 three.js 开发的吗？

**回答**：
“我们核心渲染层确实是基于 Three.js 的。但数字孪生不是只画个3D模型，它包括：
- **场景编辑器**：我们基于 Three.js 封装了一套可视化的场景配置器，让非开发人员也能拖拽设备模型、绑定数据点位。
- **实时数据映射**：通过 WebSocket 接收 IoT 数据，再用着色器（Shader）或模型材质变化动态反映设备状态（如温度红变蓝）。
- **性能优化**：复杂工厂模型面数百万级，必须做 LOD（细节层次）、实例化渲染（InstancedMesh）、视锥体裁剪。否则页面会卡死。
- **为什么不直接用 Unity/虚幻**：因为要在浏览器里无缝集成我们的 Vue 后台系统，Three.js 给了我们完全的 Web 生态控制权。”

### 3. webpack 是做什么用的？（本质回答）

**本质**：一个**静态模块打包器**。它从入口文件开始，递归构建依赖图，然后用 loader 把各种资源转成有效模块，再用 plugin 在构建生命周期中做额外处理，最终输出优化的 bundle。

**追问陷阱**：别只说“打包工具”，要能说出它的核心流程：
1. 初始化 Compiler，读取配置。
2. 从 Entry 开始，调用 Loader 翻译模块。
3. 递归解析依赖，生成模块依赖图。
4. 根据不同 Module 类型，用不同 Template 输出到 Chunk。
5. 通过 Plugin 在编译各阶段做钩子拦截（如 `HtmlWebpackPlugin`）。

**为什么后来用 Vite**：在开发阶段，Vite 利用浏览器原生 ES Module，免去打包，冷启动极快，HMR 几乎瞬间。Webpack 需全量打包再启动 dev server，大项目会慢。

### 4. 多轮对话的上下文是怎么保存的？

这道题是你的 AI 项目核心，必须答出**前后端协调**和**长窗口管理**。

**回答**：
“多轮对话的上下文由服务端负责持久化，前端只负责维护会话标识和前端展示状态。

1. **会话标识**：用户进入对话就生成一个 `conversationId`，存 Pinia + localStorage。刷新后通过 ID 恢复。
2. **消息历史**：每次对话，前端只传本次新消息和 conversationId，服务端从数据库/缓存里把该会话之前的 N 条消息取出来，组成完整的 `messages` 数组发给模型。上下文永远在后端拼接。
3. **前端缓存**：我们前端会缓存最近的消息列表在 store 里，用于快速展示。刷新时，先展示本地缓存的快照，同时请求服务端最新的历史做校对。
4. **长上下文管理**：模型有 token 限制。我们会实现**滑动窗口**或**摘要压缩**：
   - 滑动窗口：只保留最近 K 轮。
   - 摘要压缩：把早期对话总结成一段摘要，作为“长期记忆”贴到系统 prompt 里，既省 token 又不丢信息。
5. **工程化**：我们封装了一个 `useConversation` 组合函数，统一处理以上逻辑。流式输出时，新消息对象动态 append 到列表，页面直接用 `v-for` 渲染，响应式一直保持。”

**追问**：如果用户在生成过程中刷新页面，怎么恢复上下文？
**回答**（见上一批流式输出问题）：我们通过会话 ID 向服务端查询该会话最新消息列表，同时前端有 `sessionStorage` 做增量兜底。

### 6. 菜单权限怎么做？keep-alive 页面缓存怎么做？

这两道题经常一起问，因为路由 + 权限 + 缓存是后台系统的筋骨。

#### 菜单权限（本质）
**本质**：**基于角色的访问控制（RBAC），通过路由配置的 meta 信息与用户权限数组动态匹配，生成可访问路由表**。

**核心流程**：
1. 用户登录 → 后端返回权限标识数组 `['dashboard', 'order:list', 'order:edit']`。
2. 前端路由表分两部分：
   - **公共路由**（登录、404）。
   - **异步路由**（需要权限的路由），每个路由 `meta.roles` 或 `meta.perms`。
3. 在 `router.beforeEach` 里调用 `store.dispatch('permission/generateRoutes')`，用 `filter` 或递归函数过滤出用户有权访问的路由。
4. 用 `router.addRoute` 动态注册这些路由。
5. 侧边栏菜单根据权限路由表递归渲染，支持多级嵌套、外链、显示/隐藏。

**关键点**：
- 动态路由要按后端返回的结构生成，不仅仅是扁平数组。
- 刷新后要重新生成路由，所以权限数据要缓存（可存 localStorage 做首屏加速，但真实校验必须再调接口）。
- 按钮权限：除了路由，还用自定义指令 `v-permission="'order:edit'"` 控制按钮显示。

#### keep-alive 页面缓存
**本质**：保留组件实例和 DOM 在内存中，避免重复渲染。

**与路由配合**：
- 在需要缓存的 `<router-view>` 外包裹 `<keep-alive :include="cachedViews">`。
- `cachedViews` 是一个保存在 Vuex/Pinia 中的字符串数组，存需要缓存的组件名称。
- 使用技巧：
  1. 页面 `name` 必须与路由 `name` 一致，否则无法匹配。
  2. 提供关闭标签页的功能：当用户关闭一个标签页，从 `cachedViews` 数组中移除该组件名，同时 `$destroy` 或调用 `beforeRouteLeave` 清理。
  3. 用 `activated` / `deactivated` 生命周期来处理恢复/离开时的逻辑（比如重新拉取数据、清除定时器）。

**追问陷阱**：
- “keep-alive 和 beforeRouteEnter 谁先触发？”  
  答：首次进入时 `beforeRouteEnter` -> `created` -> `mounted` -> `activated`。从缓存恢复时，只会触发 `activated` 和 `beforeRouteEnter`（如果该守卫是写在组件内且没有用 `next` 模式，实际需要小心）。

### 7. 页面嵌套如何开发？微前端 vs iframe

这是架构题，考察你有没有真正在项目里权衡过方案。

#### 本质
- **iframe**：浏览器原生提供的**硬隔离沙箱**。
- **微前端**：在 JavaScript 层面实现**路由分发和应用生命周期管理**，多个子应用共存于同一文档流。

#### 对比维度

| 维度 | iframe | 微前端（qiankun / Micro-app） |
|------|--------|------------------------------|
| **隔离性** | 完美沙箱：CSS/JS/执行上下文完全隔离 | 软隔离：需处理 CSS 污染、JS 沙箱（快照/Proxy） |
| **通信** | 通过 `postMessage`，序列化成本高，消息格式需约定 | 可以共享全局状态（如主应用的 store），通信自然 |
| **性能** | 每个 iframe 有独立的渲染进程，内存占用大，加载慢 | 子应用资源按需加载，与主应用共享进程，性能较好 |
| **用户体验** | 页面跳转生硬，无法共用弹窗、消息提示等 | 无缝切换，UI 一致，可共用全局组件 |
| **公共依赖复用** | 不能复用，每个 iframe 都会加载自己的 Vue/React | 可通过 `externals` 抽取公共依赖，减少重复加载 |
| **适用场景** | 需要绝对安全的第三方页面嵌入、不同技术栈且完全不信任的子应用 | 同一个产品体系下的多个业务模块，需要统一体验 |

#### 我的项目实践
“我们内部后台是多团队协作的，不同业务线有自己的仓库。最初用 iframe，但遇到了三个致命问题：
1. 弹窗被 iframe 边界截断，全局 Loading 无法统一。
2. 登录态同步复杂，需要不断 postMessage。
3. 浏览器内存飙升严重，十几个 iframe 标签页让系统变慢。

后来迁移到 **qiankun 微前端**。我主导做了几件事：
- 设计了全局的**应用注册规范**和生命周期钩子。
- 抽离了公共的 Vue、VueRouter、Element UI 到主应用，子应用 external 掉，总体积减少 60%。
- 利用 qiankun 的沙箱，配合 **Shadow DOM** 解决了 CSS 污染。
- 对于极个别需要强隔离的旧系统，我们仍保留 iframe 作为‘子应用容器’，通过 qiankun 统一管理。

最终：开发体验独立，用户体验统一，内存下降 40%。”
