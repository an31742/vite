### 1. 做了哪些性能优化？（全景回答）

#### 本质
性能优化的本质是“**在完整用户体验链路上，消除或提前每一环节的等待时间**”。它不是点状优化，而是**从网络层、构建层、渲染层、运行时层、感知层**的全链路体系。

#### 我实际做过的分层优化

**① 网络层**
- **HTTP/2**：全站启用，多路复用减少连接数。
- **资源压缩**：开启 Gzip/Brotli，图片用 WebP 格式，大图懒加载。
- **CDN**：静态资源、字体、第三方库全部走 CDN，并设置强缓存（`max-age=31536000`）。
- **预连接**：对关键第三方域名做 `dns-prefetch`、`preconnect`。

**② 构建层**
- **代码分割 (Code Splitting)**：路由懒加载，`() => import()` 按需加载路由组件。
- **Tree Shaking**：保证 ES Module 不被打包，每次构建后分析 bundle 构成（`webpack-bundle-analyzer`）。
- **第三方库外置**：通过 externals 把 Vue/AG-Grid/ECharts 抽出，子应用共享，避免重复加载。
- **Vite 迁移**：部分项目从 Webpack 切到 Vite，利用 ESM 和 esbuild 提升冷启动和 HMR 速度。

**③ 渲染层（首屏核心）**
- **SSR + 流式渲染**：首屏服务端直出，减少白屏时间。
- **骨架屏**：对无法 SSR 的页面用骨架屏维持感知。
- **图片与字体优化**：字体切片、`font-display: swap` 防止文字不可见。
- **关键 CSS 内联**：首屏必要样式内联到 HTML。

**④ 运行时层**
- **虚拟滚动**：AG-Grid 的 infinite 模式、自研穿梭框和大列表都用虚拟滚动。
- **大数据切断响应式**：用 `shallowRef` 代替 `reactive` 存万级表格数据，避免深度劫持。
- **Web Worker**：把数据格式化、差集/交集等计算丢到 Worker，主线程只负责渲染。
- **防抖节流**：resize、scroll、输入框都做防抖或节流。
- **内存管理**：大屏销毁时 dispose ECharts 实例、解绑事件，避免泄漏。

**⑤ 感知层**
- **Loading 状态精细化**：用骨架屏、进度条、过渡动画转移等待注意。
- **乐观更新**：某些操作（如点赞、状态切换）先更新 UI，再发请求，失败时回滚。

#### 高级表述
“面试官，我理解性能优化不是修修补补，而是一套工程机制。我们项目里我主导做了三件事：**建立性能基线和监控**（Lighthouse CI + 自研埋点看 FCP/LCP）；**制定性能预算**，比如首屏 JS 不超过 300KB；**技术方案选型时前置考虑性能成本**，比如选 AG-Grid 前就规划好虚拟滚动和切断响应式。这样才能可持续，而不是出了问题再救火。”

### 2. 首屏优化如何做？（从原理到指标）

#### 本质
首屏优化的本质是**缩短从用户输入 URL 到首屏内容完整渲染的“关键渲染路径”长度**。

#### 核心流程（按时间线拆解）
**① DNS 解析** → **② TCP 连接** → **③ TLS 握手** → **④ HTTP 请求/响应** → **⑤ HTML 解析** → **⑥ 首次渲染 (FCP)** → **⑦ JS/CSS 加载执行** → **⑧ 最大内容绘制 (LCP)** → **⑨ 可交互 (TTI)**

#### 具体优化手段

**🔹 网络与资源加速**
- **DNS 预解析**：`<link rel="dns-prefetch">`
- **预连接**：`<link rel="preconnect">`
- **HTTP/2 + CDN + 动态加速**
- **强缓存与协商缓存**：静态资源指纹化 + 长缓存

**🔹 资源体积与数量削减**
- **代码分割**：路由懒加载，`webpackChunkName` 分组
- **Tree Shaking + Dead Code Elimination**
- **图片优化**：WebP 格式、压缩、响应式图片 (`srcset`)
- **字体子集化**：只抽用到的字符

**🔹 关键渲染路径优化**
- **关键 CSS 内联**：首屏样式直接写 `<style>` 里，非关键 CSS 异步加载（`media="print"` 切换）
- **JS 延迟加载**：`async` 或 `defer`，首屏不需要的脚本延后执行
- **SSR 或 SSG**：直接返回有内容 HTML，缩短 FCP
- **骨架屏 / loading 动画**：让用户感知首屏在“发生”

**🔹 预加载与预渲染**
- **Preload**：`<link rel="preload">` 提前加载关键资源（如字体、核心 JS）
- **Prefetch**：空闲时加载下一步可能用到的资源
- **Prerender**：预渲染下一页，用于强预测场景

**🔹 运行时优化**
- **虚拟列表**：首屏只渲染可视区 DOM
- **数据预取**：在路由守卫里提前请求数据，或使用 `async setup`
- **避免渲染阻塞**：不把大数据计算放主线程初始化，用 Worker 或延迟计算

#### 项目实战案例
“在我们大屏项目中，首屏是 3D 模型 + 数据图表，初次加载要 6 秒。我做了这些事：
1. **大屏 SSR 不合适**，就改用**骨架屏**加**加载进度条**维持感知。
2. 3D 模型用 **glTF 格式**压缩、**Draco 压缩**，并做**渐进式加载**：先出低模再替换高模。
3. 首屏可见图表预先用静态占位，等 JS 加载完再激活 ECharts。
4. 用 `preload` 提前下载核心 JS bundle 和关键模型纹理。
最终首屏可交互时间从 6 秒降到 2.8 秒。”

### 3. 处理大量数据如何优化保证性能：JS 处理差集、交集、并集、补集

#### 本质
JS 中集合运算的性能瓶颈在于**时间复杂度**和**主线程占用**。数组方法如 `filter`、`includes` 套用会导致 O(n²) 复杂度，且大数据下阻塞渲染。

#### 优化策略

**① 使用 Set/Map 降复杂度**
- Set 的 `has` 是 O(1)，Map 同理。将“查找集合”转为 Set。
- **交集**：遍历集合 A，用 Set 检查是否在 B 中。
  ```js
  const setB = new Set(B);
  const intersection = A.filter(x => setB.has(x));
  ```
- **差集 (A - B)**：`A.filter(x => !setB.has(x))`
- **并集**：`[...new Set([...A, ...B])]`
- **补集 (全集U中A的补集)**：`U.filter(x => !setA.has(x))`
复杂度从 O(n*m) 降为 O(n+m)。

**② 大规模数据放 Web Worker**
如果数组都十万条，Set 构造和遍历依然耗时（主线程 100ms+ 会掉帧）。应该：
- 将原始数据通过 `postMessage` 传给 Worker。
- 在 Worker 内使用上述 Set 方法计算。
- 结果回传。主线程不卡顿。

**③ 增量处理与分片 (Time Slicing)**
如果必须主线程算，用 `requestIdleCallback` 或 `scheduler` API 分片处理：
```js
function processInChunks(data, batchSize, callback) {
  let index = 0;
  function chunk() {
    const end = Math.min(index + batchSize, data.length);
    for (; index < end; index++) {
      // 处理 data[index]
    }
    if (index < data.length) {
      requestIdleCallback(chunk);
    } else {
      callback();
    }
  }
  requestIdleCallback(chunk);
}
```

**④ 结合 AG-Grid 处理表格数据**
“在我们中后台里，两个列表有 5 万条数据，需要快速对比出差集。我直接在后端或 Web Worker 里用 Set 做差集，前端只展示结果。如果必须前端，就用 `shallowRef` 存数组，计算时在 Worker 里进行，返回普通数组直接赋值，跳过 Vue 响应式转换，用户感受不到任何卡顿。”

#### 追问：Object 和 Map 在性能上的区别？
- Map 对频繁增删和任意键查找更优，键可以是任意类型，且保持插入顺序。
- Object 的 key 只能是 string/symbol，大量属性时可能出现原型链问题，且无内置 size。
- 在用作查找表时，Map 的 `get` 比 Object 属性访问略快，尤其在动态键场景。

### 4. 如何处理跨域？

#### 本质
跨域的本质是**浏览器的同源策略**：协议、域名、端口任一不同，即禁止读取响应。这是一种安全隔离，但给合理的前后端分离开发带来麻烦。

#### 解决方案全景（按场景选型）

| 方案 | 适用场景 | 原理 |
|------|---------|------|
| **CORS** | 常规前后端分离，可控后端 | 服务端设置 `Access-Control-Allow-Origin` 等响应头 |
| **代理** | 开发环境 | Vite `server.proxy` / Webpack `devServer.proxy` 将请求代理到后端，绕过浏览器同源 |
| **Nginx 反向代理** | 生产环境统一域名 | 前端请求同域名，Nginx 把 `/api` 转发到真实后端 |
| **JSONP** | 老式 GET 跨域 | 动态创建 `<script>` 标签，利用它不受同源限制；仅支持 GET，安全性差 |
| **postMessage** | 不同窗口/iframe 通信 | 双向跨文档通信，可用于不同域间的消息传递 |
| **WebSocket** | 实时双向通信 | 不受同源策略限制，但会检查 Origin，服务端可拒绝 |
| **document.domain + iframe** | 同主域子域交互 | 设置相同主域，绕过限制（过时） |

#### 我的项目实践
“我们微前端主应用和子应用在不同子域时，API 请求统一走 CORS。后端配置了允许的 Origin 列表，并带上 `Access-Control-Allow-Credentials: true`，前端请求设置 `withCredentials: true` 传递 cookie。开发时 Vite 的 proxy 直接代理到后端服务，实现无缝调试。

复杂一点的场景，比如我们内部嵌入一个第三方的监控页面（不同域），用 `postMessage` 做父子页面的令牌传递和高度同步，安全且灵活。

JSONP 我们已完全不用，原因很明确：只支持 GET、没有错误处理、有 XSS 风险。”

#### 追问：CORS 预检请求 (Preflight) 什么情况触发？
- 当请求是**非简单请求**时触发：比如用了 `PUT/DELETE`，或 Content-Type 是 `application/json`，或带自定义 Header。
- 浏览器先发一个 `OPTIONS` 请求，征得服务器同意后，再发真正的请求。

#### 追问：跨域时如何携带 Cookie？
- 服务端设 `Access-Control-Allow-Credentials: true`，且 `Access-Control-Allow-Origin` 不能为 `*`，必须指定具体域名。
- 前端在 Fetch 或 XHR 里开启 `credentials: 'include'`。
