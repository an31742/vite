### 1. Vue2 和 Vue3 的区别（架构视角）

#### 本质
本质是 **“响应式实现机制的革命”和“编程组织范式的升级”**。不是修修补补，而是底层从 `Object.defineProperty` 切换到 `Proxy`，上层从 Options API 演进到 Composition API。

#### 核心区别分层

**① 响应式系统（最根本的变化）**
- Vue2：基于 `Object.defineProperty` 劫持已有属性。致命缺陷：无法检测属性添加/删除，数组需重写 7 个方法，初始化时要递归遍历整个对象。
- Vue3：基于 `Proxy` 代理整个对象。可以拦截任意属性的增删改查，数组直接用原生方法，惰性响应式（读到才劫持），性能更好，内存更省。

**② API 组织范式**
- Vue2：Options API。按选项类型分割（data、methods、computed），逻辑关注点分散，mixins 有命名冲突和来源不清问题。
- Vue3：Composition API。按逻辑关注点组织，一个 `setup` 函数可将相关逻辑聚合，组合函数（Composables）天然解耦，逻辑复用清晰。

**③ 组件根节点**
- Vue2：必须有单根节点（底层虚拟 DOM diff 依赖单根）。
- Vue3：支持多根节点（Fragment），编译器用 Fragment 类型包裹，patch 时直接处理子节点数组。

**④ 编译优化**
- Vue2：模板编译后生成 `render` 函数，diff 时全量比较。
- Vue3：静态节点提升（hoistStatic）、动态节点标记（PatchFlags）、靶向更新。只对比动态部分，跳过静态节点，性能大幅提升。

**⑤ TypeScript 支持**
- Vue2：TypeScript 支持是后加的，类型推导困难。
- Vue3：从源码层用 TypeScript 重写，天然类型推导。

**⑥ 生命周期变化**
- `destroyed` → `unmounted`，`beforeDestroy` → `beforeUnmount`。
- 新增 `setup()` 替代 `created` 和 `beforeCreate`。

**⑦ 架构变化**
- Vue3 采用 Monorepo 架构，模块可独立引入（如只用 `@vue/reactivity`）。
- Tree-shaking 友好，未用到的特性不会被打包。

#### 项目实践中的感受
“迁移到 Vue3 后，最爽的不是性能，而是 **Composition API 让代码逻辑可复用、可测试**。比如跨项目的权限校验、AI 对话流式处理，我写成 Composables，到处引用。同时 Proxy 的响应式让数组操作变得更自然，不再有 Vue2 里 `this.$set` 那种‘补丁式’代码。”

### 2. 组织树渲染和 AG-Grid 如何做性能优化

#### 本质
两者共同的性能瓶颈是 **“大量 DOM 节点创建、深度遍历开销、以及 Vue 响应式对巨量数据的劫持”**。

#### 组织树（Tree）渲染优化

**① 虚拟滚动 + 节点懒加载**
- 组织树不是全量展开。只渲染可见区域的节点，使用自研或基于 `vue-virtual-scroller` 的树形虚拟列表。
- 节点展开时才加载子节点数据，而非初始化时递归加载整棵树。

**② 响应式剥离**
- 树节点数据通常是只读的，不需要 Vue 深度劫持。
- 用 `shallowRef` 或 `markRaw` 标记树数据，避免递归代理。

**③ 扁平化存储与渲染**
- 后台存树用邻接表（id + parentId），前端加载后转为扁平 Map，渲染时按需从 Map 取子节点。
- 避免深层嵌套对象的递归查找。

**④ 节点复用**
- 如果树节点类型统一，使用 `v-for` + `key` 确保 Diff 高效。
- 对复杂自定义节点，用函数式组件减少实例化开销。

#### AG-Grid 渲染优化（结合你的实战）

**① 启用客户端无限滚动或服务端模式**
- `rowModelType: 'infinite'`：AG-Grid 只创建视口内行的 DOM。
- `serverSide`：服务端分页和排序，前端仅渲染当前页。

**② 切断 Vue 响应式**
- 这就是你从 12 秒优化到 1.2 秒的核心。
- 用 `shallowRef` 或 `markRaw` 包裹传给 `rowData` 的数组。
- 如果某个单元格需要响应式，单独对那一个字段做 `ref`。

**③ 列定义优化**
- 列定义对象用 `Object.freeze` 冻结，避免 Vue 做不必要的劫持。
- 自定义 `cellRenderer` 用函数式组件或纯字符串，而非完整 Vue 组件。

**④ 事件与回调优化**
- AG-Grid 的回调频繁触发（如 `onCellValueChanged`），用防抖/节流包装。
- 事件处理器在 `onMounted` 创建，`onUnmounted` 销毁，避免内存泄漏。

**⑤ 数据流优化**
- 大数据更新时，用 `api.setRowData()` 而非直接改响应式变量。
- 批量操作前后调用 `api.batchUpdateRowData()`，减少内部刷新次数。

#### 综合案例回答
“组织树和表格看似不同，但优化思想一致：**少创建 DOM、少劫持数据、多利用原生 API**。我们一个权限管理系统，组织树 5000 节点，AG-Grid 20000 行。组织树用扁平化存储 + 懒加载，AG-Grid 用 infinite 模式 + `shallowRef` 存数据，首帧加载都在 1 秒内。”

### 3. Webpack 打包有泄露源码的危险，如何处理

#### 本质
泄露的本质是 **Source Map 文件暴露在生产环境，导致浏览器开发者工具可直接查看原始源码，包括注释、文件路径、业务逻辑**。

#### 防护方案（从简单到深入）

**① 生产环境关闭 Source Map**
```js
// webpack.config.js
module.exports = {
  mode: 'production',
  devtool: false, // 或 'hidden-source-map'
};
```
- 完全关闭：`devtool: false`。
- 隐藏但保留给内部分析：`devtool: 'hidden-source-map'`。这样生成 `.map` 文件，但不在 JS 文件末尾添加 `//# sourceMappingURL`，浏览器不会自动加载。内部分析时手动上传到 Sentry 等监控平台。

**② 使用 `nosources-source-map`**
- 显示错误堆栈的行号，但不展示具体源代码内容。
- 适合需要线上排错但不想完全暴露源码的场景。

**③ Source Map 文件不上传 CDN**
- 生成 `.map` 文件后，只存储在 CI 服务器或内部存储，不随静态资源部署。
- CI 构建时将 map 文件上传到 Sentry 或专用错误监控平台，然后删除，不入制品库。

**④ 代码压缩与混淆**
- Webpack 生产模式默认用 Terser 压缩混淆，变量名变为单字母，结构打乱，降低可读性。
- 但这只是增加阅读难度，不能替代 Source Map 管理。

**⑤ 内容安全策略（CSP）**
- 响应头设置 `Content-Security-Policy: default-src 'self'`，限制资源来源。
- 虽然不能直接防源码泄露，但可限制 XSS 和代码注入。

**⑥ 代码分割与授权访问**
- 核心逻辑拆到独立 chunk，结合后端权限校验，未授权无法加载。
- 这不是防泄露的根本，但增加攻击复杂度。

**⑦ Git 泄露预防**
- `.git` 目录不要部署到生产环境。
- `.gitignore` 加好，不在代码仓库中存 `.map` 文件或密钥。

#### 项目落地
“我们 CI 流程里，生产构建用的是 `hidden-source-map`，生成的 map 文件直接上传到自建的错误监控服务，不在服务器落盘。DevTools 网络面板里看不到 map 加载请求，断点调试完全不可能。”

### 4. 用了哪些 AI 插件，AI 能提效多少

#### 插件与工具
- **GitHub Copilot / Copilot Chat**：代码补全、注释生成。
- **Cursor**：更智能的上下文理解，重构能力。
- **Claude / ChatGPT 网页**：方案设计、排错、文档生成。
- **内部知识库机器人**：对接组件库文档和历史 Bug 解决方案。

#### 提效量化（结合实际）
- **编写工具函数 / 单元测试**：提效 50%+，因为 AI 非常擅长固定模式。
- **写 ECharts 配置 / 表格列定义**：从文档查找拷贝 → 一句话描述生成，提效 60%。
- **文档撰写**：AI 生成初稿，我修订，节省 50% 时间。
- **代码审查**：AI 找空值检查、错误处理遗漏，准确率约 70%，审查速度提升 30%。
- **方案调研**：复杂场景的初版方案生成，能帮我在 1 小时内覆盖 3-4 个方向，以前要半天。

#### 核心态度
“AI 是我手里的一个强力提效器，但我始终把握：**核心业务逻辑必须我自己理解并写出**；AI 生成代码必须 Code Review 后才能合并；当 AI 产出的方案我无法完全解释时，坚决不用。提升的是效率，不是替代我的判断力。”

### 5. 如果任务比较多应该如何处理

#### 本质
本质是 **“有限资源下的异步优先级调度”**——这也是事件循环教会我们的：区分宏任务和微任务，区分紧急与重要。

#### 我的处理框架

**① 梳理与分类（Inbox 清零）**
- 所有任务列出来，按**紧急度**和**重要性**四象限分类。
- 区分“必须今天完成”和“可以明天开始”。

**② 优先级排序**
- 紧急且重要：立即做。
- 重要不紧急：计划时间块深度做。
- 紧急不重要：快速处理或委托。
- 不紧急不重要：最后做或拒绝。

**③ 时间分片（Time Slicing）**
- 和 React Fiber 一样道理，大任务拆成小片，每完成一个小片休息或处理突发事件。
- 一天划出 2-3 个深度工作块，期间关闭通知，专注复杂问题。

**④ 并行与串行决策**
- 独立任务并行推进（如同 JavaScript 的 `Promise.all`）。
- 有依赖的任务串行安排，提前说清阻塞点。

**⑤ 沟通透明化**
- 如果任务超量，第一时间和上级同步风险，给出**可交付物清单**和**时间评估**。
- 不说“我太忙了做不完”，而是说“在 A、B、C 里，我现在只能保质完成两项，C 是否可以下周或由同事支援”。

**⑥ 防打断机制**
- 记录打断频率，若频繁则推动流程改进（如设答疑时段、完善文档）。
- 和低优先级请求者友好协商：“下午 4 点我给你准确答复”。

**⑦ 复盘调优**
- 每周回顾实际耗时和预估偏差，提高自己时间评估准确度。
- 这也是对自己“性能 profiler”的持续优化。

#### 面试回答精简版
“遇到任务多，我先分类梳理，和需求方沟通真实优先级，把大任务拆成小任务用时间分片推进。关键是我的时间开销数据透明，一旦预测交付有风险，第一时间和上级对齐，不在最后时刻爆发。这就像事件循环，不阻塞主线程，关键路径要优先保障。”
