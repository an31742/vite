### 1. 盒模型

#### 本质
本质是“**CSS中每个元素都是一个矩形盒子，由内容、内边距、边框、外边距四部分组成，浏览器根据 `box-sizing` 属性决定width/height的计算范围**”。

#### 为什么这样设计
- **解决问题**：提供一种一致的空间计算规则，让布局可控。
- **两种模式**：
  - **标准盒模型** (`box-sizing: content-box`)：`width` = 内容宽度，实际占据空间 = width + padding + border。
  - **怪异/IE盒模型** (`box-sizing: border-box`)：`width` = 内容 + padding + border 的总和，实际内容宽度会被压缩。

#### 核心追问与陷阱
- **追问**：一个 `div` 设 `width: 100px; padding: 10px; border: 5px solid;`，标准模式下实际占用宽度是多少？
  - 标准答案：100 + 10*2 + 5*2 = 130px。
- **陷阱**：分不清 `margin` 是否包含在 `width` 里。永远记住 `margin` 不属于盒子的尺寸，它是外部距离，不参与 `box-sizing` 计算。
- **工程化**：全局重置 `*, *::before, *::after { box-sizing: border-box; }` 是现代项目的标配，因为 `border-box` 更符合直觉，“设置多大就是多大”。

### 2. 双等号和三等号区别

#### 本质
本质是“**在比较值时，是否允许隐式类型转换**”。
- `==` 允许，会触发一套复杂的抽象相等比较算法。
- `===` 严格禁止，类型不同直接返回 false。

#### 核心陷阱
- `null == undefined` 是 `true`，`null === undefined` 是 `false`。这是 `==` 的特例。
- `[] == ![]` 为 true？这属于代码混淆题，实际项目绝不会这么写。回答这类问题时要表明：“我理解有类型转换规则，但在团队规范里，我们禁止使用 `==`，统一用 `===`，用 ESLint `eqeqeq` 规则卡死。”

### 3. typeof 与引用类型复制（理解为你问的“type of 复制”）

我推测你想问的是：`typeof` 在判断引用类型时的局限，以及复制引用类型时要注意什么。

#### 本质
- `typeof` 能准确判断基本类型（除了 `null`），但对引用类型一律返回 `"object"`（函数返回 `"function"`）。
- 复制引用类型时，直接赋值只是复制了地址，指向同一块堆内存。

#### 项目陷阱
- 用 `typeof` 判断数组、对象、`null` 都是 `"object"`，要做精确判断应使用：
  - `Array.isArray()`
  - `Object.prototype.toString.call(x)` 返回 `[object Type]`
  - `x === null`
- 复制一个引用类型，如果只做浅拷贝，深层修改会污染源数据。在 Vue 项目中，如果你把 store 里的对象直接赋值给另一个变量并修改，会违反单向数据流，导致调试灾难。

### 4. 后端返回超大整数，前端精度缺失怎么处理？

#### 本质
本质是“**JavaScript 的 Number 类型基于 IEEE 754 双精度浮点数，安全整数范围为 `[-2^53+1, 2^53-1]`，超过此范围的整数在 JSON 解析时发生舍入误差**”。

#### 现象
比如后端返回 `{ "id": 12345678901234567890 }`，前端 `JSON.parse` 后，这个 id 变成 `12345678901234567000`，末位丢失。

#### 解决方案（按推荐度）

1. **与后端约定，将大整数用字符串传输**（最佳）。  
   接口返回 `{ "id": "12345678901234567890" }`，前端直接当作字符串使用，不参与运算。这是代价最小、最可靠的方案。

2. **使用 json-bigint 等库**，在解析时把特定字段转为 BigInt 或字符串。  
   ```js
   const JSONbig = require('json-bigint')({ storeAsString: true });
   const data = JSONbig.parse(responseText);
   ```

3. **后端使用 Snowflake 等算法生成的 ID，如果主键确实只用于标识，强烈建议后端直接返回 string**。前端不应背这个锅。

#### 项目真实案例
“在我们中后台项目中，对接一个Java微服务，雪花ID过长。初期没注意，结果列表里ID全不一样，前端拿错误ID去请求详情，各种诡异。最后和后端达成规范，所有 long 类型 ID 在序列化时全局转成 string，前端零改动，彻底消灭了精度问题。”

### 5. Promise 和 async/await 的区别

#### 本质
- **Promise**：是异步操作的**状态容器**，通过 `.then()` 链式调用。
- **async/await**：是 Promise 的**语法糖**，用同步写法写异步，底层仍然是 Promise 和微任务。

#### 关键区别（面试要答出这个深度）

1. **错误处理**：
   - Promise 用 `.catch()`，错误容易遗漏。
   - async/await 可直接用 `try/catch`，代码结构更清晰。

2. **可读性**：
   - Promise 链式调用，复杂逻辑仍然可能形成“长链”。
   - async/await 看起来像同步代码，阅读和维护成本低。

3. **调试**：
   - async/await 可以像同步代码一样打断点，Promise 链在异步回调里跳转比较痛苦。

4. **返回值**：
   - `async` 函数自动返回 Promise，非 Promise 值会被包装。

5. **执行顺序微妙差异**：
   ```js
   async function foo() {
     console.log(1);
     await console.log(2);
     console.log(3);
   }
   foo();
   console.log(4);
   // 输出 1,2,4,3
   ```
   `await` 后面的语句相当于 `.then()` 回调，被放入微任务。

#### 面试高级表述
“我们在项目里已经全面转向 async/await 了。它让我们可以像写同步逻辑一样处理异步流程，配合 `for...of` 可以优雅地顺序执行多个请求。但我也清楚，async/await 不能完全替代 Promise，像 `Promise.all`、`Promise.race` 并发场景还是得用。我把它们看作组合技。”
