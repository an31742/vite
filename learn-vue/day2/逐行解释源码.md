### 一、为什么需要三层依赖结构？

你提出的问题：“既然已经有 `targetMap`，为什么还要 `depsMap` 和 `deps`？” 这个设计是为了**精确追踪哪个对象（target）的哪个属性（key）被哪些副作用（effect）依赖**。

#### 1. 三层结构的职责

```js
targetMap: WeakMap<target, depsMap>
depsMap:   Map<key, deps>
deps:      Set<effect>
```

| 层级 | 存储结构 | 键 | 值 | 作用 |
|------|---------|----|----|------|
| **第一层** `targetMap` | `WeakMap` | 响应式对象（如 `state`） | `depsMap` | 区分不同的响应式对象 |
| **第二层** `depsMap` | `Map` | 属性名（如 `"count"`） | `deps` | 区分同一个对象的不同属性 |
| **第三层** `deps` | `Set` | — | 依赖该属性的所有 `effect` | 收集需要重新执行的副作用 |

#### 2. 为什么不能只用一层？

假设把所有依赖全放在一个全局 `Set` 中：

```js
// ❌ 错误设计
globalDeps = new Set();

function track(target, key) {
  if (activeEffect) globalDeps.add(activeEffect);
}

function trigger(target, key) {
  globalDeps.forEach(e => e()); // 一个属性变了，所有 effect 都执行
}
```

修改 `state.count` 时，连只依赖 `state.name` 的 `effect` 也会被执行，违背了 Vue 的**细粒度更新**原则。

**三层结构就是为了精准配对**：修改 `state.count` → 只触发依赖 `state.count` 的那些 `effect`。

#### 3. 逐层解析

- **第一层 `targetMap`（WeakMap）**  
  `targetMap: { state → depsMap }`  
  键：响应式对象（如 `state`），值：一个 `Map`。  
  使用 `WeakMap` 的好处：当对象不再被引用时，垃圾回收可自动清理该对象的所有依赖记录，防止内存泄漏。

- **第二层 `depsMap`（Map）**  
  `depsMap: { "count" → deps, "name" → deps }`  
  键：属性名，值：一个 `Set`，存储所有依赖该属性的 `effect`。

- **第三层 `deps`（Set）**  
  `deps: { effectFn1, effectFn2 }`  
  存储包装后的 `effectFn`（即我们定义的箭头函数）。`Set` 保证不重复收集。当属性变化时，遍历此 `Set` 执行所有 `effect`。

#### 4. 完整流程示例

```js
const state = reactive({ count: 0, name: 'Vue' })

effect(() => { console.log(state.count) })   // track(state, 'count')
effect(() => { console.log(state.name) })    // track(state, 'name')
```

执行后 `targetMap` 结构：

```
targetMap (WeakMap) = {
  state → Map {
    "count" → Set { effectFn1 },
    "name"  → Set { effectFn2 }
  }
}
```

修改 `state.count = 1` 只会执行 `effectFn1`，修改 `state.name = 'Vue3'` 只会执行 `effectFn2`。这就是精准更新的秘密。

### 二、手动实现响应式系统（逐行解释）

以下代码构建了一个最小可用的响应式引擎。建议新建 `effect-demo.js` 边写边运行。

#### 1. 全局变量与存储

```js
let activeEffect = null           // ① 当前正在执行的副作用函数
const targetMap = new WeakMap()   // ② 全局依赖映射
```

**解释**：
- ① `activeEffect`：用于标记当前正在收集依赖的 effect 函数。初始 `null` 表示没有 effect 在执行。
- ② `targetMap`：`WeakMap` 结构，存放所有响应式对象及其依赖表。键为原始对象，值为 `Map`（`depsMap`）。

#### 2. 依赖收集 `track`

```js
function track (target, key) {
  if (!activeEffect) return            // ① 无活跃 effect，不收集

  let depsMap = targetMap.get(target)  // ② 获取对象的依赖表
  if (!depsMap) {
    targetMap.set(target, (depsMap = new Map())) // ③ 不存在则创建
  }

  let deps = depsMap.get(key)          // ④ 获取属性的依赖集合
  if (!deps) {
    depsMap.set(key, (deps = new Set())) // ⑤ 不存在则创建
  }
  deps.add(activeEffect)               // ⑥ 将当前 effect 加入集合
}
```

**逐行解释**：
- ① 只有当 `activeEffect` 不为 `null` 时才收集依赖。普通代码（非 effect 内）读取属性时不会触发收集。
- ② 尝试从全局 `targetMap` 获取该 `target` 对应的 `depsMap`。
- ③ 如果该对象还没有依赖表，新建一个 `Map` 并关联到 `targetMap`。括号中的 `(depsMap = new Map())` 同时完成赋值和设置。
- ④ 从 `depsMap` 获取该属性 `key` 对应的依赖集合 `deps`。
- ⑤ 如果该属性还没有依赖集合，新建一个 `Set` 并关联。
- ⑥ 将当前活跃的 `activeEffect` 添加到该属性的依赖集合中，完成收集。

#### 3. 触发更新 `trigger`

```js
function trigger (target, key) {
  const depsMap = targetMap.get(target)   // ① 获取对象依赖表
  if (!depsMap) return                    // ② 无依赖表则退出
  const deps = depsMap.get(key)           // ③ 获取属性的依赖集合

  if (deps) {
    deps.forEach(effect => effect())      // ④ 执行所有依赖的 effect
  }
}
```

**逐行解释**：
- ① 从全局映射中取出该 `target` 的依赖表。
- ② 如果没有，说明没有任何 effect 依赖此对象的任何属性，直接返回。
- ③ 取出具体属性的依赖集合。
- ④ 遍历集合并执行每一个 effect（即我们包装的 `effectFn`）。执行会重新运行用户逻辑，获取最新值。

#### 4. 响应式代理 `reactive`

```js
function reactive (target) {
  return new Proxy(target, {
    get (target, key, receiver) {
      track(target, key)                      // ① 收集依赖
      return Reflect.get(target, key, receiver) // ② 返回原始值
    },
    set (target, key, value, receiver) {
      const oldValue = target[key]             // ③ 记录旧值
      const result = Reflect.set(target, key, value, receiver) // ④ 设置新值
      if (oldValue !== value) {                // ⑤ 比较新旧值
        trigger(target, key)                   // ⑥ 触发更新
      }
      return result;
    }
  })
}
```

**逐行解释**：
- ① 每次读取属性（如 `state.count`）时，调用 `track` 收集当前 `activeEffect` 为依赖。
- ② `Reflect.get` 安全地从原始对象中获取属性值并返回。`receiver` 保证 getter 中 `this` 指向代理对象。
- ③ 在赋值前先保存原始对象上的当前值（`oldValue`）。
- ④ `Reflect.set` 将新值写入原始对象，返回布尔值表示是否成功。此时数据已更新。
- ⑤ 只有新旧值不同时才触发更新，避免无意义的重新执行。
- ⑥ 调用 `trigger`，找出所有依赖该属性的 effect 并重新执行。

#### 5. 副作用函数 `effect`

```js
function effect (fn) {
  const effectFn = () => {        // ① 创建包装函数
    activeEffect = effectFn;      // ② 将自己设为当前活跃 effect
    fn();                         // ③ 执行用户函数（触发 get → track）
    activeEffect = null;          // ④ 执行完毕，清除标记
  }
  effectFn();                     // ⑤ 立即执行一次，完成首次依赖收集
}
```

**逐行解释**：
- ① 定义一个箭头函数 `effectFn`，它是一个闭包，能访问外部 `fn`。
- ② 执行用户函数前，将全局 `activeEffect` 指向 `effectFn` 自身，这样后续的属性读取都能通过 `track` 收集到它。
- ③ 执行用户传入的 `fn`（如 `() => console.log(state.count)`）。此时所有对响应式数据的读取都会触发 `track`，将 `effectFn` 注册为依赖。
- ④ 执行完毕后重置 `activeEffect` 为 `null`，表示当前没有 effect 在运行，避免不必要的依赖收集。
- ⑤ 立即调用 `effectFn()`，完成首次收集和用户逻辑执行。

### 四、进阶扩展：`stop` 与 `cleanup` 的作用

在实际 Vue3 中，`effect` 还有 `stop` 和 `cleanup` 机制，是面试高频考点。

#### 1. 支持 `stop`

```js
function effect(fn) {
  const effectFn = () => {
    activeEffect = effectFn;
    fn();
    activeEffect = null;
  };
  effectFn.deps = [];   // 反向记录该 effect 被哪些 deps 收集
  effectFn();
}

// 修改 track，让 effect 反向记录 deps
function track(target, key) {
  if (!activeEffect) return;
  // ... 创建 depsMap, deps 的逻辑 ...
  deps.add(activeEffect);
  activeEffect.deps.push(deps);   // 反向收集
}

function stop(effectFn) {
  effectFn.deps.forEach(dep => dep.delete(effectFn));
}
```

#### 2. 为什么需要 `cleanup`

每次 effect 重新运行时，可能会读取不同的属性。如果不清理旧的依赖，已不再读取的属性发生变化时仍会触发该 effect，造成“幽灵依赖”。

Vue3 的 `ReactiveEffect` 在每次执行前会调用 `cleanupEffect` 清空旧的依赖关系，然后重新收集，保证依赖的准确性。

### 五、完整心智模型

```
state.count = 1
     ↓
Proxy set 陷阱
     ↓
Reflect.set(target, 'count', 1)   ← 真实值已修改
     ↓
oldValue !== value ？ 是 → trigger(state, 'count')
     ↓
从 targetMap 找到 depsMap，再找到 deps (Set)
     ↓
遍历 deps，执行每个 effectFn
     ↓
effectFn 内部再次读取 state.count，拿到新值 1
     ↓
用户逻辑（console.log）输出新值
```

### 六、今日核心问答

**Q：`get` 中的 `key` 是什么？**  
A：被访问的属性名字符串，如 `state.count` 中 `key` 为 `"count"`。

**Q：`Reflect.get/set` 做了什么？**  
A：安全地读写对象属性，并能正确传递 `receiver`（代理对象），避免 `this` 指向错误。

**Q：`set` 何时触发？**  
A：给代理对象的属性赋值时触发，`state.count = 1` 即触发 `set` 陷阱。

**Q：`value` 和 `oldValue` 的关系？**  
A：`value` 是要设置的新值，`oldValue` 是设置前原始对象的值。比较它们可以避免无意义的更新。

**Q：`trigger` 怎么更新值？**  
A：`trigger` 不修改值，它只是重新执行依赖该属性的 effect；effect 内部再次读取最新数据，从而获取新值。

**Q：`effectFn` 闭包的作用？**  
A：包装用户函数，在执行前将自身设为 `activeEffect`，使 `track` 能够收集，执行后清除标记。闭包特性使 `fn` 在将来触发时仍能正确执行。

**Q：`activeEffect = null` 后依赖是否失效？**  
A：不会。依赖收集发生在 `fn()` 执行期间，已经记录到 `targetMap` 中。后续 `trigger` 会再次设置 `activeEffect` 并重新收集，保证依赖一直有效。
