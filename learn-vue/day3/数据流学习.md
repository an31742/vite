
### 一、核心困惑解答：为什么 `oldValue = newValue` 就能记录旧值？

你的困惑完全正常，因为**这里的“旧值”并不是靠什么神奇的历史回溯，而是靠一个持续存在的闭包变量 `oldValue`**。

看这段 `watch` 的简化代码：

```js
function watch(source, cb) {
  let oldValue, newValue

  const scheduler = () => {
    newValue = getter()           // 拿到这次变化后的最新值
    cb(newValue, oldValue)       // 把“本次新值”和“上次存下来的 oldValue”传给回调
    oldValue = newValue          // 把这次的新值保存起来，变成下一次的“旧值”
  }

  // ... 创建 effect，scheduler 被保存在 effectFn 上
}
```

关键在于：`oldValue` 是**声明在 `watch` 函数作用域内的变量，被 `scheduler` 闭包持有**。它不会因为函数执行结束而消失。

用时间线举个例子：

1. **初始化**：执行 `oldValue = effectFn()` → `oldValue = 0`
2. **第一次变化** `state.count = 10`：
   - 调度器运行：`newValue = getter()` → `10`
   - 调用 `cb(10, 0)` ← 这时的 `oldValue` 就是之前保存的 `0`
   - 执行 `oldValue = newValue` → `oldValue` 被更新为 `10`
3. **第二次变化** `state.count = 20`：
   - 调度器再次运行：`newValue = getter()` → `20`
   - 调用 `cb(20, 10)` ← 此时的 `oldValue` 已经是上次更新的 `10` 了，完美作为“旧值”
   - 执行 `oldValue = newValue` → `oldValue` 更新为 `20`

**所以本质上，`oldValue` 就是一个“接力棒”，每次跑完一圈就传给下一次使用。** 不是通过什么魔法回溯历史，而是利用闭包持久存储上一个值。

### 二、`watchEffect` 数据流全景

`watchEffect` 等于一个**自动收集依赖，依赖变了就重新执行回调**的 `effect`。

```
【创建阶段】
watchEffect(callback)
  → 创建 effectFn
  → effectFn 内：
       cleanup(effectFn)            // 首次无旧依赖
       activeEffect = effectFn      // 把自己标记为活跃
       callback()                   // 执行用户回调
         → 内部读取 state.count     // 触发 track，把 effectFn 收集为依赖
       activeEffect = null
  → 立即执行 effectFn()（因为默认 lazy: false）

【变化阶段】
state.count = 1
  → Proxy set → trigger(state, 'count')
  → 找到 deps 中的 effectFn，发现没有 scheduler
  → 直接调用 effectFn()
       → cleanup(effectFn)          // 清除上一次依赖（本次会重新收集）
       → activeEffect = effectFn
       → callback()                  // 重新执行，拿到的就是新值 1
       → activeEffect = null
```

**一句话**：只要 effectFn 再跑一次，内部读取的值就是最新的，`watchEffect` 的回调自然拿到新值，无需新旧值对比。

### 三、`watch` 数据流全景

`watch` 的核心是把**“追踪依赖”**和**“执行业务回调”**拆开了。

```
【创建阶段】
watch(getter, cb)
  → 创建 effectFn，配置 { lazy: true, scheduler }
  → 传给 effect 的 fn 是 () => getter()（只取值，不含业务）
  → 因为 lazy，effectFn 不会自动执行
  → 手动调用 oldValue = effectFn()
       → 内部执行 getter()，读取 state.count，触发 track 收集 effectFn
       → 返回当前值，赋值给 oldValue（如 0）
  → 用户的 cb 至今没有执行

【变化阶段】
state.count = 1
  → trigger 找到 effectFn，发现有 scheduler
  → 不执行 effectFn 本身，而是执行 scheduler()
       → 如果之前注册了清理函数，先执行
       → newValue = getter()         // 直接调用 getter，拿到新值 1
       → cb(newValue, oldValue)      // 调用用户回调，oldValue 是上一次存的值（0）
       → oldValue = newValue         // 保存为下次使用的旧值（1）
```

**为什么在 scheduler 里用 `getter()` 而不是 `effectFn()`？**
- `effectFn()` 会重新设置 `activeEffect`，引发新一轮依赖收集，并可能触发 `cleanup` 等不必要操作。
- 如果 `effectFn()` 内部再不小心修改数据，会造成死循环。
- 直接运行 `getter()` 只是读取最新值，简洁安全。

### 四、`watchEffect` vs `watch` 一览表

| 特性 | `watchEffect` | `watch` |
|------|----------------|----------|
| **回调执行** | 立即执行一次，之后每次依赖变化再执行 | 默认不执行，只在监听源变化时执行 |
| **依赖收集** | 自动收集回调内所有响应式依赖 | 仅收集 getter 中读取的依赖，非常明确 |
| **是否拿到旧值** | 不能（直接重跑回调，无历史记录） | 能（通过闭包保存 `oldValue`） |
| **调度器** | 无，直接重跑 effectFn | 有 `scheduler`，在依赖变化时拦截 effectFn |
| **典型场景** | 简单的“数据变了我要做点事” | 需要精确监听某个值并对比变化 |

### 五、如何验证你完全理解了？

用下面的代码自己跑一遍，在脑海或控制台里追踪每一步的变量变化：

```js
const state = reactive({ count: 0 })

watchEffect(() => {
  console.log('watchEffect 看到:', state.count)
})

watch(
  () => state.count,
  (newVal, oldVal) => {
    console.log(`watch 看到: ${oldVal} -> ${newVal}`)
  }
)

console.log('开始修改...')
state.count = 1
state.count = 2
```

**预期输出 + 内部变量轨迹**：

| 步骤 | 输出 | 内部状态 |
|------|------|----------|
| 初始化 | `watchEffect 看到: 0` | watchEffect 的 effectFn 被收集到 count 的依赖；watch 的 oldValue = 0，其 effectFn 也被收集 |
| `state.count = 1` | `watchEffect 看到: 1`<br>`watch 看到: 0 -> 1` | watchEffect 的 effectFn 直接重跑；watch 的 scheduler 运行，getter 返回 1，调用 cb(1, 0)，然后 oldValue = 1 |
| `state.count = 2` | `watchEffect 看到: 2`<br>`watch 看到: 1 -> 2` | 同上逻辑，watch 的 oldValue 这次已经是 1 了 |
