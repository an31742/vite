好问题！`cb` 本身并不具备缓存能力，它只是一个普通函数，**真正“记住”旧值的是 `watch` 内部那个被闭包捕获的变量 `oldValue`**。

---

### 1. 变量放在哪里？

```js
function watch(source, cb) {
  let oldValue, newValue    // ← 这两个变量属于 watch 函数的作用域

  const scheduler = () => {
    newValue = getter()     // 读取最新值
    cb(newValue, oldValue)  // 把 oldValue 传给回调
    oldValue = newValue     // 把当前新值保存为下一次的旧值
  }

  // 创建 effect，内部引用了 scheduler
  const effectFn = effect(() => getter(), { lazy: true, scheduler })
  // ...
}
```

- `oldValue` 是声明在 `watch` 函数内部的局部变量。
- `scheduler` 是一个箭头函数，它被定义在 `watch` 内部，所以它能访问外部作用域的 `oldValue`、`newValue` 等变量——这就是**闭包**。
- 只要 `scheduler` 还存在（被保存在 effectFn 上，而 effectFn 又被引用），`watch` 的整个作用域就不会被垃圾回收，`oldValue` 就一直活着。

---

### 2. 整个生命周期

| 阶段 | 发生的事情 | oldValue 的状态 |
|------|-----------|------------------|
| `watch` 调用 | 创建 `oldValue`、`newValue` 变量 | `undefined` |
| 初始化 (`oldValue = effectFn()`) | 手动执行 getter，拿到当前值 `0` | `0` |
| 第一次变化 | scheduler 运行：<br>`newValue = getter()` → `10`<br>`cb(10, 0)` ← 使用闭包中的 `oldValue`<br>`oldValue = 10` | `10` |
| 第二次变化 | scheduler 再次运行：<br>`newValue = getter()` → `20`<br>`cb(20, 10)` ← 闭包中的 `oldValue` 现在是 `10`<br>`oldValue = 20` | `20` |
| 第三次变化 | ... | 以此类推 |

每一次 scheduler 执行时：
- **读取** `oldValue` 就是读取闭包中保存的**上一次的值**。
- **写入** `oldValue = newValue` 就是把本次的新值存进去，作为下一次的旧值。

---

### 3. 打个比方

闭包就像你在银行开了一个**保险箱**（`watch` 的作用域），里面放了一张纸条写着 `oldValue`。  
调度器是一把**钥匙**，每次数据变化时你用这把钥匙打开保险箱，看一眼纸条上的数字（旧值），然后执行回调，最后把纸条上的数字更新成最新的数字。  
下次再打开时，纸条上已经是上次的数字了。

所以不是 `cb` 有记忆，而是 **闭包变量 `oldValue` 替你记住了“上一次的值”**。

---

### 4. 为什么 `cb` 不需要缓存？

因为 `cb` 只是一个**通知者**，它不需要记住任何东西。它只在被调用时接收两个参数：
- `newValue`（刚刚从 getter 里拿到的当前值）
- `oldValue`（从闭包变量里读到的上一次值）

然后它做自己的事（打印、发请求等），做完就结束了。记忆的责任完全交给 `oldValue` 变量。

---

**总结**：  
- `cb` 不缓存旧值。
- `oldValue` 是闭包中的变量，生命周期与 `watch` 作用域绑定。
- 每次 scheduler 执行时，`oldValue` 就是上次保存的值，执行完后再更新为新值。
- 这利用了 JavaScript 闭包特性，让函数“记住”它诞生时的环境变量。

你现在已经掌握了 Vue 响应式系统中最精妙的闭包用法，接下来学 `computed` 时，你会发现它同样是用闭包变量 `_value` 来做缓存的，思路一模一样。
