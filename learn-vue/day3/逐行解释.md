## 一、`effect(fn, options)` 逐行解释

```js
function effect (fn, options = {}) {
  const { lazy = false, scheduler = null } = options
  const effectFn = () => {
    cleanup(effectFn)        // ① 清理自己的旧依赖
    activeEffect = effectFn  // ② 标记为当前活跃的副作用
    const result = fn()      // ③ 执行用户函数（getter 或副作用）
    activeEffect = null      // ④ 清除活跃标记
    return result            // ⑤ 返回 fn 的执行结果
  }

  effectFn.deps = []         // ⑥ 初始化反向收集数组
  effectFn.scheduler = scheduler  // ⑦ 挂载调度器

  if (!lazy) {
    effectFn()               // ⑧ 非懒加载时立即执行
  }
  return effectFn            // ⑨ 返回包装函数，供外部 stop 使用
}
```

### 逐行解析

- **`const { lazy = false, scheduler = null } = options`**  
  解构选项：`lazy` 控制是否立即执行，`scheduler` 是自定义调度函数。

- **`const effectFn = () => { ... }`**  
  创建一个包装函数，形成闭包。这个函数可以被多次调用。

- **① `cleanup(effectFn)`**  
  调用 `cleanup` 将自己从所有依赖集合中移除，并清空 `effectFn.deps` 数组。  
  **目的**：确保每次重新执行时，只收集本次真正读取的属性的依赖，防止“幽灵依赖”。

- **② `activeEffect = effectFn`**  
  将全局变量 `activeEffect` 指向当前副作用，这样在 `fn` 执行期间，任何响应式属性的读取都会通过 `track` 把这个 `effectFn` 收集为依赖。

- **③ `const result = fn()`**  
  执行用户传入的函数。  
  - 如果是 `effect(() => console.log(state.count))`，`fn` 就是那个打印函数，返回值是 `undefined`。  
  - 如果是 `watch` 中的 getter，`fn` 就是 `() => state.count`，返回值是 `count` 的当前值。  
  执行完毕后，依赖收集完成。

- **④ `activeEffect = null`**  
  清除活跃标记，表示没有副作用在运行，避免非 effect 内的读取被错误收集。

- **⑤ `return result`**  
  将 `fn()` 的返回值传递出去。  
  **为什么要 `return result` 而不是直接返回 `fn`？**  
  因为我们需要的是 **执行结果**（比如当前值），而不是函数本身。  
  在 `watch` 中，我们需要拿到 getter 的返回值作为旧值/新值，所以必须返回执行结果。

- **⑥ `effectFn.deps = []`**  
  初始化反向收集数组，用于存储所有包含该 `effectFn` 的 `deps` 集合（Set），以便 `cleanup` 和 `stop` 能高效清理。

- **⑦ `effectFn.scheduler = scheduler`**  
  将调度器挂载到 `effectFn` 上，供 `trigger` 使用。当依赖变化时，`trigger` 会检查 `effect.scheduler` 是否存在，若存在则调用它而不是直接执行 `effect`。

- **⑧ `if (!lazy) { effectFn() }`**  
  默认 `lazy: false`，会立即执行一次，完成首次依赖收集。如果 `lazy: true`（如 `watch`），则只创建不执行，由外部决定何时调用。

- **⑨ `return effectFn`**  
  返回副作用函数本身，方便外部手动 `stop` 或获取值。

## 二、`watch(source, cb, options)` 逐行解释

```js
function watch (source, cb, options = {}) {
  let oldValue, newValue
  let cleanupFn = null
  const onCleanup = (fn) => { cleanupFn = fn }

  const getter = typeof source === 'function' ? source : () => source

  const effectFn = effect(
    () => getter(),       // 传给 effect 的 fn
    {
      lazy: true,
      scheduler: () => {
        if (cleanupFn) cleanupFn()        // 执行上次的清理
        newValue = getter()               // 获取新值
        cb(newValue, oldValue, onCleanup) // 执行用户回调
        oldValue = newValue               // 更新旧值
      }
    }
  )

  if (options.immediate) {
    newValue = effectFn()                  // 立即执行，获取当前值
    cb(newValue, undefined, onCleanup)     // 以 undefined 作为旧值调用回调
    oldValue = newValue
  } else {
    oldValue = effectFn()                  // 只取值，不触发回调
  }

  return () => stop(effectFn)
}
```

### 逐行解析

- **`let oldValue, newValue`**  
  声明变量，用于存储新旧值。

- **`let cleanupFn = null`**  
  存储用户通过 `onCleanup` 注册的清理函数。

- **`const onCleanup = (fn) => { cleanupFn = fn }`**  
  提供给用户注册清理函数的方法，类似 Vue 的 `onCleanup`。

- **`const getter = typeof source === 'function' ? source : () => source`**  
  统一将监听源转换为 getter 函数。如果 `source` 是函数（如 `() => state.count`），直接使用；否则（如 `ref` 或 `reactive` 对象）包装成 `() => source`。

- **`const effectFn = effect(...)`**  
  创建一个 `lazy: true` 的 effect，并传入自定义 `scheduler`。  
  - 传给 `effect` 的 `fn` 是 `() => getter()`。这个函数只负责**读取响应式数据并返回值**，不包含用户回调。  
  - `lazy: true` 让 effect 不会立即执行，而是由外部控制。

- **`scheduler` 函数**  
  这是 `watch` 的核心。当依赖的属性变化时，`trigger` 不会直接执行 `effectFn`，而是调用 `scheduler`。  
  ```js
  scheduler: () => {
    if (cleanupFn) cleanupFn()        // 1. 先执行上次回调注册的清理
    newValue = getter()               // 2. 重新执行 getter 获取新值
    cb(newValue, oldValue, onCleanup) // 3. 调用用户回调，传递新旧值和 onCleanup
    oldValue = newValue               // 4. 更新旧值，供下次使用
  }
  ```

- **`if (options.immediate) { ... } else { ... }`**  
  - 如果 `immediate: true`，手动调用 `effectFn()` 获取当前值作为 `newValue`，并立即以 `undefined` 作为旧值执行回调，然后让 `oldValue` 等于当前值（此时 `oldValue` 和 `newValue` 确实相同，但这是第一次调用，之后的调用就会不同）。  
  - 否则，只调用 `effectFn()` 获取初始值并赋给 `oldValue`，不触发回调。

- **`return () => stop(effectFn)`**  
  返回一个函数，调用它可以停止监听。

## 三、回答你的疑问

### Q1: `cleanup(effectFn)` 只是自己清理自己吗？

**不是“自己清理自己”，而是“清除自己与外部依赖集合的双向联系”。**

具体来说，`cleanup` 做了两件事：
1. 遍历 `effectFn.deps` 数组（里面存放的是多个 `Set`），从每个 `Set` 中删除自己（`dep.delete(effectFn)`）。
2. 清空 `effectFn.deps` 数组。

所以它并不是销毁 `effectFn` 本身，而是**切断它与所有响应式属性之间的依赖关系**，为下一次执行时重新收集全新的依赖做准备。

如果没有这一步，上次执行中依赖但本次不再使用的属性发生改变，仍然会错误地触发该 effect，造成“幽灵依赖”。

---

### Q2: `const result = fn(); return result` 直接返回 `fn` 不可以吗？

**不可以。**  
因为 `fn` 是函数本身（`() => getter()`），而我们需要的是**这个函数的执行结果**（即 getter 返回的当前值）。

- 如果返回 `fn`，那外部得到的就是一个函数，每次想获取值都得再调用它，且无法直接得到初始值。
- 而 `return result` 返回的是 `fn()` 的返回值（比如数字 `1`），对于 `watch` 来说，这就得到了被监听属性的当前值，可以用作 `oldValue` 或 `newValue`。

这里的 `result` 是 `fn()` 的返回值，而不是 `fn`。这样设计才能让 `effectFn()` 既完成依赖收集，又能顺便拿到值。

---

### Q3: 在 `watch` 的 `scheduler` 中，为什么用 `getter()` 而不是 `effectFn()`？

你代码中注释 `// 修复：直接调用 getter 而不是 effectFn()`，这是非常关键的修正。

如果使用 `newValue = effectFn()`，会引发**死循环**：
1. `effectFn()` 内部会重新设置 `activeEffect = effectFn`，然后执行 `fn()`（即 `() => getter()`）。
2. 执行 `getter()` 时会读取响应式属性，触发 `track`，而 `track` 又可能因为某些原因触发 `trigger`？
3. 更致命的是：`effectFn()` 执行后，如果内部再次修改了数据，或者由于 `effectFn` 本身被调用，可能再次触发同一个 `scheduler`，导致无限递归。

直接使用 `getter()` 可以**避免重新运行完整的 effect 流程**（即避免重新设置 `activeEffect`、调用 `cleanup` 等），只是纯粹地读取最新值。这既获取了新值，又不会触发额外的副作用或死循环。

另外，直接调用 `getter()` 不会进行依赖收集，因为此时 `activeEffect` 已经为 `null`（在 `scheduler` 调用时，`activeEffect` 是 `null` 或可能是其他的，但肯定不是当前 `effectFn`）。这样就不会在获取新值的时候误将其他 effect 收集进来。

---

### Q4: `cb(newValue, oldValue, onCleanup)` 是做什么的？

`cb` 是用户调用 `watch` 时传入的**回调函数**（callback）。

它在这里被调用，目的是**通知用户监听的值发生了变化**，并传递：
- `newValue`：变化后的新值。
- `oldValue`：变化前的旧值。
- `onCleanup`：一个函数，允许用户在回调内部注册清理逻辑（比如清除定时器、取消请求等）。这个清理函数会在下次 `scheduler` 执行前（或 watch 停止时）被调用。

例如：
```js
watch(() => state.count, (newVal, oldVal, onCleanup) => {
  console.log(`从 ${oldVal} 变为 ${newVal}`)
  const timer = setTimeout(() => { /* ... */ })
  onCleanup(() => clearTimeout(timer))
})
```

---

### Q5: `oldValue = newValue` 这样 oldValue 和 newValue 不会一样吗？

在执行完 `oldValue = newValue` 后，**确实会暂时相等**，但这是为了**下一次变化时能正确提供旧值**。

- 第一次变化：`scheduler` 内先取得 `newValue`（比如 `10`），此时 `oldValue` 还是之前的 `5`。执行 `cb(10, 5, onCleanup)` 后，将 `oldValue` 更新为 `10`。这样当下一次变化发生时，`oldValue` 就是正确的旧值 `10`，而 `newValue` 会是新值 `15`。
- 对于 `immediate` 的情况：第一次执行后 `oldValue` 被赋值为 `newValue`，它们当然相等，但这只发生一次。当后续数据变化再次触发 `scheduler` 时，`oldValue` 就是上次的值，`newValue` 是本次新值，两者就会不同。

所以这个操作是为了“滚动更新”旧值，保证下一次回调能拿到正确的 `oldValue`。

## 四、完整工作流示例

```js
const state = reactive({ count: 0 })

watch(
  () => state.count,
  (newVal, oldVal, onCleanup) => {
    console.log(`old: ${oldVal}, new: ${newVal}`)
  }
)

state.count = 1
state.count = 2
```

**执行流程**：

1. 创建 watch：调用 `effect(() => getter(), { lazy: true, scheduler })`，不立即执行。
2. 由于 `immediate` 未设置，执行 `oldValue = effectFn()`：这会运行 `effectFn`，内部执行 `getter()` 读取 `state.count`，收集依赖（`effectFn` 被加入 `count` 的 deps）。返回 `0`，`oldValue = 0`。
3. `state.count = 1`：触发 setter → `trigger(count)` → 发现 `effectFn` 有 `scheduler`，调用 `scheduler`。
   - 执行 `cleanupFn`（目前为空）
   - `newValue = getter()` → 返回 `1`
   - `cb(1, 0, onCleanup)` → 打印 `old: 0, new: 1`
   - `oldValue = 1`
4. `state.count = 2`：再次触发 `scheduler`。
   - `newValue = getter()` → `2`
   - `cb(2, 1, onCleanup)` → 打印 `old: 1, new: 2`
   - `oldValue = 2`

完美闭环。

### 今日核心总结

- `effect` 是整个响应式系统的**执行单元**，负责依赖收集和重新触发。
- `cleanup` 不是消灭自己，而是**切断旧的依赖关系**，为重新收集做准备。
- `return result` 是为了**获取 fn 的执行结果**，而非函数本身。
- `watch` 的 `scheduler` 中直接使用 `getter()` 而不是 `effectFn()`，是为了**避免死循环和错误收集**。
- `cb` 调用传递新旧值和清理函数。
- `oldValue = newValue` 是为了**滚动记录**，保证下一次回调拿到正确的旧值。

你现在的实现已经非常接近 Vue 3 的实际逻辑，理解了这些细节就掌握了响应式系统最核心的设计智慧。
